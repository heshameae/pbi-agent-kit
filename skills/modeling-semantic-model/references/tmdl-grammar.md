# TMDL Grammar Reference

Canonical TMDL syntax rules, file layout, indentation, emission gotchas, and enum value sets.

**Primary sources:** microsoft/skills-for-fabric (tmdl-authoring-guide) · data-goblin/dg1-pbip (tmdl/SKILL.md, object-properties.md) · ruiromano/powerbi-agentic-plugins (TMDL.md)

Concrete table/column/measure names in this reference are syntax illustrations only. For production writes, replace every identifier with fields from the deterministic planner, validated spec, or live model inventory. Do not copy sample names into a user model.

---

## TMDL File Layout

| File | Contents | Location |
|---|---|---|
| `model.tmdl` | Model config, `ref table` entries, query groups, annotations | `definition/` |
| `database.tmdl` | Compatibility level, model ID | `definition/` |
| `relationships.tmdl` | All relationships between tables | `definition/` |
| `expressions.tmdl` | Shared M expressions and parameters | `definition/` |
| `functions.tmdl` | DAX user-defined functions | `definition/` |
| `tables/<Name>.tmdl` | Table: columns, measures, hierarchies, partitions | `definition/tables/` |
| `roles/<RoleName>.tmdl` | One file per security role (RLS filters, OLS) | `definition/roles/` |
| `perspectives/<Name>.tmdl` | One file per perspective | `definition/perspectives/` |
| `cultures/<locale>.tmdl` | Linguistic metadata and translations | `definition/cultures/` |
| `dataSources.tmdl` | Legacy data source definitions | `definition/` |

## Syntax Rules

### Indentation

TMDL uses **tab-based indentation** — depth equals nesting level. Never spaces.

```tmdl
table Product                              // depth 0: top-level declaration
    lineageTag: abc-123                    // depth 1: table property

    measure '# Products' =                // depth 1: measure declaration
            COUNTROWS (                    // depth 3: DAX expression body
                VALUES ( Product[Name] )
            )
        formatString: #,##0               // depth 2: measure property
        displayFolder: Measures
        lineageTag: def-456

    column 'Product Name'                  // depth 1: column declaration
        dataType: string                   // depth 2: column property
        summarizeBy: none
        sourceColumn: Product Name
```

### Indentation Depth Summary

**Rule: a multi-line DAX body is always 2 levels deeper than its enclosing declaration.**

| Context | Depth | Tabs |
|---|---|---|
| Top-level declaration (`table`, `relationship`, `expression`) | 0 | 0 |
| Table properties; column/measure/hierarchy declarations | 1 | 1 |
| Column/measure properties; hierarchy levels | 2 | 2 |
| DAX body for measure/column declared at depth 1 (inside table) | 3 | 3 |
| `calculationItem` inside `calculationGroup` (depth 1) | 2 | 2 |
| DAX body for `calculationItem` at depth 2 | 4 | 4 |
| DAX body for top-level `function` at depth 0 | 2 | 2 |

### Descriptions (`///`)

Triple-slash sets the `Description` property on the **next** declaration. Native TMDL syntax (not Tabular Editor extension).

```tmdl
/// Count of distinct products in the current filter context.
measure '# Products' =
        COUNTROWS ( VALUES ( Product[Product Name] ) )
    formatString: #,##0
```

**Rules:**
- `///` must be immediately followed by a declaration on the next line — no blank line
- Multiple `///` lines concatenate into a single description
- Applies to: `measure`, `column`, `table`, `hierarchy`, `level`, `relationship`

**Wrong:**
```tmdl
// blank line between /// and declaration
/// This is a description.

measure 'My Measure' = 1    // WRONG
```

### Comments (`//`)

`//` is a regular comment with no semantic effect. Supported inside DAX expression blocks and M expressions. **Not supported** at the TMDL file level outside of code blocks — use `///` for descriptions.

### Name Quoting

Use single quotes around names that contain: spaces, dots, equals, colons, single quotes, or names starting with a digit. Do NOT quote simple identifiers (`Product`, `Date`, `_Measures`).

## Emission Gotchas

These are the silent failure modes for TMDL generation tools:

| Rule | Detail |
|---|---|
| **No hand-written `lineageTag`** | Auto-generated by Power BI. Adding manually causes GUID collisions — Desktop fails to open |
| **Measures before columns** | Every table definition must list all measures first, then columns |
| **No `dataType` on measures** | Inferred from DAX expression; adding it is invalid |
| **No `//` at TMDL file level** | Only inside M or DAX code blocks |
| **Triple-backtick for multi-line DAX** | Use either triple-backtick block OR indented block; never mix |
| **Leave `PBI_*` annotations** | `PBI_FormatHint`, `PBI_Id`, etc. are Power BI internal; do not add, modify, or remove |
| **`///` immediately precedes declaration** | No blank line between description and declaration |
| **M-expression/table name collision** | Shared M expressions and tables share a namespace; duplicate names fail with `'duplicate member <name>'`. Append ` Query` to M expressions |
| **Object names in single quotes only when needed** | Quote only when names contain spaces or special chars; never quote simple names |

## Measure Definitions

### Multi-Line DAX — Triple-Backtick Block

```tmdl
measure Percentage = ```
        VAR _Total = CALCULATE( SUM ( 'Table'[Quantitative] ), REMOVEFILTERS ( ) )
        RETURN
        DIVIDE ( SUM ( 'Table'[Quantitative] ), _Total )
        ```
    formatString: 0.0%;-0.0%;0.0%
    lineageTag: abc-123
```

### Multi-Line DAX — Indented Block

```tmdl
measure '<Actuals MTD Measure>' =
        CALCULATE (
            [Actuals],
            CALCULATETABLE (
                DATESMTD ( '<DateTable>'[<DateKeyColumn>] ),
                '<DateTable>'[<ScopeFlagColumn>]
            )
        )
    formatString: #,##0
    displayFolder: <DisplayFolder>
    lineageTag: <ExistingOrGeneratedLineageTag>
```

### Dynamic Format String

```tmdl
measure '<Variance Percent Measure>' =
        DIVIDE ( [Actuals] - [Target], [Target] )
    displayFolder: <DisplayFolder>

    formatStringDefinition =
            IF ( [Variance %] >= 0, "+0.00%", "0.00%" )
```

`formatStringDefinition` replaces `formatString` when the format is computed dynamically.

## `summarizeBy` Rules

| Column type | Correct `summarizeBy` |
|---|---|
| Keys (surrogate/natural), foreign keys | `none` |
| Attributes (names, codes, types) | `none` |
| Dates | `none` |
| Boolean flags | `none` |
| Additive numeric facts (amounts, quantities) | `sum` |
| Non-additive numerics (rates, percentages) | `none` |

## Relationships

```tmdl
relationship '<FactToDateRelationshipName>'
    fromColumn: '<FactTable>'.'<FactDateColumn>'
    toColumn: '<DateTable>'.'<DateKeyColumn>'

/// Inactive — use with USERELATIONSHIP() in DAX
relationship '<FactAlternateDateRelationshipName>'
    isActive: false
    fromColumn: '<FactTable>'.'<AlternateRoleDateColumn>'
    toColumn: '<DateTable>'.'<DateKeyColumn>'
```

Key rules:
- `fromColumn` = many-side (fact)
- `toColumn` = one-side (dimension)
- Default: `crossFilteringBehavior: oneDirection`
- Set `isKey: true` on dimension PK columns; hide FK columns on fact tables

## Object Nesting Rules

| Object | Allowed Parent(s) |
|---|---|
| `column`, `measure`, `hierarchy`, `partition`, `calculationGroup` | `table` |
| `level` | `hierarchy` |
| `calculationItem` | `calculationGroup` |
| `tablePermission` | `role` |
| `columnPermission` | `tablePermission` |
| `annotation`, `extendedProperty` | any object |
| `formatStringDefinition` | `measure`, `calculationItem` |

Root-level objects (depth 0 only): `model`, `database`, `table`, `relationship`, `role`, `cultureInfo`, `perspective`, `dataSource`, `expression`, `queryGroup`, `function`

## Partition Templates

### Import (M expression)

```tmdl
partition '<ImportPartitionName>' =
    mode: import
    source = m
        let
            Source = Sql.Database("server", "db"),
            <EntityStepName> = Source{[Schema="<SchemaName>", Item="<EntityName>"]}[Data]
        in
            <EntityStepName>
```

### Direct Lake

```tmdl
partition '<DirectLakePartitionName>' = entity
    mode: directLake
    source
        entityName: <EntityName>
        expressionSource: DatabaseQuery
```

### Calculated Table

```tmdl
partition '<DateTablePartitionName>' = calculated
    mode: import
    source = CALENDAR( DATE(<ObservedStartYear>, <ObservedStartMonth>, <ObservedStartDay>), DATE(<ObservedEndYear>, <ObservedEndMonth>, <ObservedEndDay>) )
```

Use observed fact min/max dates from `pbi_model_plan_date_table.recommendedRange`; do not hardcode calendar bounds or use `TODAY()`/`NOW()` as default anchors.

## Date/Calendar Table

- Prefer an existing date table from the source over auto-generated
- Ensure contiguous date range with no gaps; full span of fact data
- Mark the Date table only through `pbi_date_table_create_governed` or `pbi_table_mark_as_date(tableName, dateColumn, facts)` after Date-table coverage proof succeeds; do not set primitive `dataCategory: Time` / `isKey` metadata directly. Import models may reject the `isKey` write; live unique/contiguous Date-key proof plus table `dataCategory: Time` is authoritative.
- Configure `sortByColumn` for month name → month number
- Disable auto-date tables in production models

## Annotations

```tmdl
annotation SummarizationSetBy = Automatic
annotation PBI_FormatHint = {"isGeneralNumber":true}
```

- Annotations are key-value pairs; indented under the object they annotate
- Do **not** add `PBI_*` annotations manually
- `annotation` is at the same depth as other properties of the parent object

## Calculation Groups

Calculation groups are special tables with a `calculationGroup` block instead of regular partitions.

```tmdl
table 'Time Intelligence'
	lineageTag: abc-123
	isHidden

	calculationGroup
		precedence: 10

		calculationItem Current = SELECTEDMEASURE()
			ordinal: 0

		calculationItem YTD = ```
				CALCULATE (
				    SELECTEDMEASURE(),
				    DATESYTD ( '<DateTable>'[<DateKeyColumn>] )
				)
				```
			ordinal: 1
			formatStringDefinition = SELECTEDMEASUREFORMATSTRING()

		calculationItem 'YoY %' = ```
				VAR _Current = SELECTEDMEASURE()
				VAR _Prior = CALCULATE ( SELECTEDMEASURE(), SAMEPERIODLASTYEAR ( '<DateTable>'[<DateKeyColumn>] ) )
				RETURN
				DIVIDE ( _Current - _Prior, _Prior )
				```
			ordinal: 5
			formatStringDefinition = "0.00%"

	column Name
		dataType: string
		lineageTag: def-456
		sourceColumn: Name
		summarizeBy: none

	partition 'Time Intelligence' = calculationGroup
		mode: import
		source
			precedence: 10
```

**Calc group rules:**
- Partition `source` type is `calculationGroup` — **not** `m` or `calculated`
- `precedence` declared at both `calculationGroup` and `partition source` levels
- `isHidden` on the table hides it from the Fields pane (recommended)
- `SELECTEDMEASUREFORMATSTRING()` for passthrough items; explicit `"0.00%"` for percentage items
- Always include a `Current` passthrough item at ordinal 0
- `formatStringDefinition` replaces `formatString` when the format is computed dynamically

## Measures-Only Table

Use a calculated table with a dummy `{1}` source as a pure measure container:

```tmdl
table __Measures
	lineageTag: abc-123

	column Value
		isHidden
		lineageTag: def-456
		isNameInferred
		sourceColumn: [Value]

	partition __Measures = calculated
		mode: import
		source = {1}
```

**Rule:** All measures in the table are declared before this dummy column. The table has no data role — it exists purely as a measure home.

## Calculated Column

```tmdl
column Profit
	dataType: decimal
	formatString: \$#,##0.00
	lineageTag: abc-123
	summarizeBy: sum
	sourceColumn: Profit

	expression = [Revenue] - [Cost]
```

**When to use calculated columns vs measures:**
- Calculated columns: row-by-row (Profit = Revenue - Cost), segmentation/categorisation, composite keys, row-level business rules, filtering on the result
- Measures: aggregations (SUM, AVERAGE), dynamic filter-context calculations, time intelligence, minimising model size

## Hierarchy Level

```tmdl
hierarchy 'Date Hierarchy'
	lineageTag: abc-123

	level Year
		lineageTag: def-456
		column: Year

	level Quarter
		lineageTag: ghi-789
		column: Quarter

	level Month
		lineageTag: jkl-012
		column: 'Month Name'
```

Levels are depth-2 objects (inside a hierarchy, which is depth-1 inside a table).

## Role Member Declaration

```tmdl
role '<RoleName>'
	modelPermission: read

	tablePermission '<SecuredTable>' = <RLSFunctionOrFilterExpression>

	member '<UserOrGroupPrincipal>'
	memberType: adPrincipal
	identityProvider: AzureAD
```

Use `USERPRINCIPALNAME()` (not `USERNAME()`) for Azure AD row-level security. `USERNAME()` returns a different value in Fabric/cloud environments.

## database.tmdl

Minimal required content — do not add extra properties:

```tmdl
database Database
	compatibilityLevel: 1605
```

`compatibilityLevel: 1605` is the Power BI Premium/Fabric default. Do not lower it.

## model.tmdl Key Properties

```tmdl
model Model
	defaultPowerBIDataSourceVersion: powerBI_V3
	discourageImplicitMeasures

	ref table <TableName1>
	ref table <TableName2>
	ref table <TableName3>

	annotation __PBI_TimeIntelligenceEnabled = 0
```

- `defaultPowerBIDataSourceVersion: powerBI_V3` — required for modern PBIP format
- `discourageImplicitMeasures` — prevents auto-aggregation on numeric columns in visuals
- `ref table <name>` entries list every table in the model (one per table file)
- `annotation __PBI_TimeIntelligenceEnabled = 0` — disables auto date/time tables (recommended in production)

## Property Ordering Convention

Within a column or measure block, properties appear in this order:

**Column:** `dataType` → `formatString` → `lineageTag` → `displayFolder` → `summarizeBy` → `isHidden` → `isKey` → `sortByColumn` → `sourceColumn` → `annotation`

**Measure:** DAX expression → `formatString` (or `formatStringDefinition`) → `lineageTag` → `displayFolder` → `annotation`

---

## `///` Anti-Pattern: Empty Separator Lines

Do not use `///` as a visual separator with an empty body. A blank `///` followed by no declaration is invalid and will silently produce a model-load error.

```tmdl
///                               -- WRONG: dangling /// with no declaration below it

measure '<MeasureName>' = SUM ( '<FactTable>'[<NumericColumn>] )
```

```tmdl
/// Total revenue in the filter context.
measure '<MeasureName>' = SUM ( '<FactTable>'[<NumericColumn>] )   -- CORRECT: /// immediately before declaration
```
