# 01 — Domain-Knowledge Adoption Decision (Review Agent A)

Consolidated, deduped adoption decision for the **domain** side of pbi-mcp-ts:
the 14 shared-knowledge skills, the BPA rule engine, the two reviewer agents,
and the data-analyst agent. Synthesized from all 20 mining findings files.

**Scope guardrails applied throughout:** Node/TS only (Python/C#/PowerShell/CLI = reference-only);
dataset-agnostic (every `'Sales'`/`'Date'`/`'Product'`/`[Region]` in source examples is a
**placeholder pattern**, never a literal to bake in). The hardcoding watchlist at the end flags
the few places a builder must actively de-couple.

### Canonical-source hierarchy (applies to every component below)
1. **microsoft/skills-for-fabric** (`skills-for-fabric-authoring.md`) — Microsoft **first-party**. Wins on TMDL conventions, INFO.* discovery, definition-envelope rules. When rules conflict, this is the tie-breaker.
2. **ruiromano/powerbi-agentic-plugins** (`powerbi-agentic-plugins.md`) — Microsoft/Rui Romano's official Claude plugin. The **deepest single DAX-perf + BPA artifact** (DAX001–021, 60-rule semantic-model BPA, report BPA). Primary for dax-patterns + BPA.
3. **data-goblin** set (`dg1-pbip.md`, `dg2-reports.md`, `dg3-semantic-models.md`, `dg4-te-fabric-hooks-root.md`) — Kurt Buhler / SQLBI lineage. **Richest report-design + naming-regex + M-folding + calc-group/RLS DAX + the two extra BPA rule sets.** dg3 mirrors #2's DAX framework (same upstream); dg2 owns report design; dg4 owns BPA expression-language + calc-group/RLS knowledge.
4. **awesome-copilot-pbi-data** (`awesome-copilot-pbi-data.md`) — strong checklist-driven STAR/RLS/MEASURES/PERFORMANCE refs + advanced DAX templates; **AdventureWorks names must be de-hardcoded.**
5. **claude-plugin-marketplace-pbi** (`claude-plugin-marketplace-pbi.md`, powerbi-master) — cleanest **TMDL→TOM grammar table** + the four-layer validation taxonomy + PBIR query-bucket map.

7. **claude-skills-data-analytics** + **antigravity-bi-bundles** — upstream **analyst methodology** (KPI contracts, audience maturity, hypothesis framing, storytelling) — the data-analyst/audience/kpi gap-fillers.

## tmdl-conventions
**Verdict:** rich content available (the single most-covered skill in the corpus).
**Canonical source(s):** skills-for-fabric `tmdl-authoring-guide.md` (first-party baseline) + powerbi-master `tmdl-syntax-reference.md` (best grammar→TOM table) + dg1 `tmdl/` (best indentation-depth table + enum value lists) + powerbi-agentic `TMDL.md` (emission gotchas). These four agree; merge into one.
**Adopt (merged):**
- **Syntax core (all four agree):** tab indentation, **one tab per level — spaces cause `TmdlFormatException`**; objects declared `<TOMtype> <Name>`; single-quote names containing space/`.`/`=`/`:`/`'` or starting with a digit, escape inner quote by doubling (`'Customer''s Name'`); reserved-keyword names must be quoted.
- **Descriptions:** `///` triple-slash **above** the object (sets TOM `Description`); must immediately precede a declaration (no blank line, never two `///` stacked); multiple `///` concatenate. `//` comments are **NOT supported** in TMDL (only inside M/DAX blocks). Never use the `description:` property.
- **Emission gotchas (powerbi-agentic + skills-for-fabric, must-haves for the writer):** never hand-write `lineageTag` on new objects (auto-generated → conflicts); **measures before columns** in table files; single-line DAX after `=`, multi-line wrapped in **triple backticks**; every measure needs `formatString` + a `///` description; never set `dataType` on a measure (inferred); leave existing `PBI_*` annotations alone, never add them manually.
- **The load-bearing indentation-depth table (dg1 — adopt verbatim):** top-level (`table`/`relationship`/`expression`) = 0 tabs; table props / column-measure-hierarchy decls = 1; column/measure props, hierarchy levels = 2; **DAX body for a measure/column inside a table = 3**; DAX body for top-level `function` = 2; `calculationItem` inside `calculationGroup` = 2; **DAX body for `calculationItem` = 4**.
- **`database.tmdl` must start with `database <name>`** then `compatibilityLevel: 1702`, `compatibilityMode: powerBI`, `language: 1033` (a bare `compatibilityLevel:` without the `database` object → `InvalidLineType` error). `model.tmdl` declares props + `ref table X`/`ref role X`/`ref cultureInfo en-US` so the engine discovers files; **`defaultPowerBIDataSourceVersion: powerBI_V3` required for Import.**
- **File layout:** `definition/{database,model,relationships,functions}.tmdl`, `definition/tables/<T>.tmdl` (hierarchies + calc groups live in the table file), `definition/roles/<Role>.tmdl`, `definition/cultures/<locale>.tmdl`, `definition/perspectives/<Name>.tmdl`, `definition.pbism`.
- **Column rules:** `dataType` required (`int64`/`decimal`/`string`/`dateTime`/`boolean`; **avoid `double`**); `sourceColumn` maps exactly to partition source; `summarizeBy: none` for non-aggregatable numerics (IDs, postal codes, years, month-numbers); `isAvailableInMdx: false` for hidden columns not used in sort/hierarchy/variation; `dataCategory` for geo; `sortByColumn` for month-name → month-number.
- **Naming (all sources agree):** tables business-friendly, **no `Fact`/`Dim` prefixes** (plural facts `Sales`, singular dims `Product`); the dimension's main column = same name as the dimension (`Product`, not `Product Name`); columns readable with spaces; measure TI-variant convention `[m]`, `[m (ly)]`, `[m (ytd)]`.
- **Format-string table (adopt as the format backend's source of truth):** currency `\$#,##0.00` (note the backslash escape; powerbi-agentic notes double-backslash in some contexts), percent `0.00%`, integer `#,##0`, decimal `#,##0.00`, thousands `#,##0,K`, millions `#,##0,,M`.
- **Relationships:** `fromColumn` = many-side (fact), `toColumn` = one-side (dim); default `crossFilteringBehavior: oneDirection`; `isActive: false` for role-playing (+ USERELATIONSHIP); integer keys, matching `dataType` both sides; `isKey: true` on dim PK (exactly one); hide FKs on fact; **no composite keys** (single surrogate int); **create relationships before dependent measures.**
- **Partition templates (skills-for-fabric — adopt as emitter templates):** Import = `partition X = m` / `mode: import` / `source = let … in …`; Direct Lake = named expression first (`AzureStorage.DataLake("https://onelake.dfs.fabric.microsoft.com/<WsId>/<LhId>",[HierarchicalNavigation=true])`) then `partition X = entity` / `mode: directLake` / `entityName`/`schemaName`/`expressionSource` (ALL partitions must be entity-source, no M; `binary` columns unsupported); Calculated = `partition X = calculated` / `mode: import` / `source = <DAX>`.
- **Object-properties enum sets (dg1 `object-properties.md` — adopt-as-is as VALID-VALUE validation sets; these are TMDL grammar enums, NOT dataset content):** `dataType` {automatic,string,int64,double,dateTime,decimal,boolean,binary,unknown,variant}; `crossFilteringBehavior` {oneDirection,bothDirections,automatic}; `securityFilteringBehavior` {oneDirection,bothDirections,none}; `summarizeBy`/`aggregateFunction` {default,none,sum,min,max,count,average,distinctCount}; `relationshipEndCardinality` {none,one,many}; `modeType` {import,directQuery,default,push,dual,directLake}; `columnType` {data,calculated,rowNumber,calculatedTableColumn}; `modelPermission` {none,read,readRefresh,refresh,administrator}.
**Mix-and-match notes:** indentation-depth table + enum lists from **dg1**; grammar→TOM keyword map from **powerbi-master**; the "no lineageTag / measures-first / `///` not `description:`" gotchas from **powerbi-agentic + skills-for-fabric**; partition emitter templates from **skills-for-fabric**. All four are mutually consistent — no real conflicts, just different depth on different facets.
**Reference-only / defer:** `bim-to-tmdl` TOM/CLI conversion; `validate_pbip.py` algorithm (port to TS, not Python); `createOrReplace` script semantics (only relevant if we emit TMDL scripts vs files).
**Conflicts resolved:** **TMDL canonical baseline = skills-for-fabric (first-party)** for *rules*, augmented by powerbi-master for the *grammar table* and dg1 for the *indentation-depth + enums*. powerbi-agentic and skills-for-fabric duplicate the gotcha list — adopt once.
**Provenance:** `dg1-pbip.md` (§ TMDL authoring + object-properties), `skills-for-fabric-authoring.md` (§ TMDL authoring + advanced features + partitions), `claude-plugin-marketplace-pbi.md` (§ TMDL→TOM grammar), `powerbi-agentic-plugins.md` (§ TMDL.md, modeling-guidelines), `dg3-semantic-models.md` (§ naming rules).

## dax-patterns
**Verdict:** rich content available (deepest skill after tmdl-conventions).
**Canonical source(s):** **powerbi-agentic `dax-performance-optimization.md` = primary** (DAX001–021 + QRY/MDL/DL tiers + FE/SE engine model). dg3 carries the byte-identical framework (same upstream — do **not** double-implement). awesome-copilot-pbi-data + powerbi-master add the **advanced template library** (ABC/Pareto, cohort, Top-N-ties, WINDOW/OFFSET/RANK, UDFs). powerbi-agentic `dax-query-guidelines.md` adds **valid-query rules** for measure validation.
**Adopt (merged):**
- **Tier-1 catalog DAX001–DAX021 (the spine — each is an anti-pattern → rewrite, dataset-agnostic).** Highest-value subset for the skill body:
  - DAX001 simple column predicates over `FILTER(Table,…)`; split `&&` into separate CALCULATE filter args; use KEEPFILTERS for non-equality.
  - DAX002 `ADDCOLUMNS`/`SUMMARIZE`+measure → `SUMMARIZECOLUMNS` (SUMMARIZECOLUMNS now fully supported in measure defs).
  - DAX003 cache repeated/context-independent expressions in `VAR`.
  - DAX004 remove duplicate/redundant CALCULATE filter predicates.
  - DAX007 `IF(cond,1,0)` → `INT(cond)` in iterators (kills callback); better, `CALCULATE(COUNTROWS, predicate)`.
  - DAX010 `FILTER(Table,…)` → `CALCULATETABLE(Table,…)` to set filter context.
  - DAX011 `DISTINCTCOUNT(col)` → `SUMX(VALUES(col),1)` when FE-bound is faster; DAX014 `DISTINCTCOUNT(PK)` → `COUNTROWS(table)`.
  - DAX012 `ALL(t)+VALUES(t[c])` → `ALLEXCEPT(t,t[c])` (only when column is actively filtered).
  - DAX018 (see Conflicts) replace `DIVIDE()` with `/` **inside row-context iterators when denominator guaranteed non-zero**.
  - DAX019/020 lift time-intelligence/dynamic filters to an outer CALCULATE so base measures stay TI-free and fuse → cross-listed in time-intelligence.
  - DAX016 `TREATAS`+`CROSSFILTER` to override relationship direction without a model change.
  - DAX021 pre-compute both aggregations at a shared key grain + `NATURALINNERJOIN` instead of TREATAS/IN re-filtering.
- **Engine "why" appendix (condense):** FE single-threaded (the usual bottleneck) vs SE multi-threaded VertiPaq (only +−×÷, GROUP BY, LEFT OUTER JOIN, SUM/COUNT/MIN/MAX/DISTINCTCOUNT); callbacks (`CallbackDataID`/`EncodeCallback`) = SE→FE row-by-row; vertical fusion (blocked by TI, per-measure filter predicates, SWITCH/IF between measures, calc-group items) vs horizontal fusion (blocked by sliced-column-not-in-groupby, runtime/table-valued filters). Core rule: "push work into SE, minimize SE scans, eliminate callbacks."
- **Valid-query rules (powerbi-agentic `dax-query-guidelines.md` — for the measure-validation step + dax-reference-check):** comments use `//` not `--`; **when defining** a measure fully-qualify `'Table'[Measure]`, **when using** it leave unqualified `[Measure]`; always fully-qualify columns (`SUM('Sales'[Amount])`), never qualify a measure with its table; always `ORDER BY` for multi-row results; CALCULATE boolean-filter restrictions (a measure/CALCULATE result must go in a VAR first; can't reference two tables in one predicate; `IN` operand must be a table var); SUMMARIZECOLUMNS for measures (no boolean filters — wrap in CALCULATETABLE), SUMMARIZE for distinct columns only (never measure exprs).
- **Core authoring rules (consensus across powerbi-agentic / awesome-copilot / agent-skills / dg3):** always use `VAR` (prefix `_`) for repeated subexpressions; **avoid IFERROR/ISERROR** (use DIVIDE + defensive modeling); don't convert BLANK→0 unless required downstream; single CALCULATE with multiple filters over nested CALCULATE; `TREATAS` over `INTERSECT`; `CROSSFILTER(...,BOTH)` in a measure over a bidirectional relationship.
- **Advanced template library (awesome-copilot + powerbi-master — adopt as parameterized templates, NOT literal):** ABC/Pareto via running-total SWITCH; `RANKX(ALL(…),[m],,DESC,DENSE)`; Top-N-with-ties via `MIN` over `TOPN`; dynamic-measure-selector via `SELECTEDVALUE`+`SWITCH` (field-parameter pattern); virtual relationships via TREATAS; parent-child via PATH/PATHITEM/PATHCONTAINS; semi-additive via LASTDATE/FIRSTDATE; cumulative total (performant REMOVEFILTERS form); %-of-parent/grand-total. 2025-2026 functions to cover: WINDOW/INDEX/OFFSET/RANK/ROWNUMBER, visual-calc FIRST/LAST/PREVIOUS/NEXT/LOOKUP, NAMEOF/TABLEOF, dynamic format strings.
- **DAX UDFs (powerbi-agentic `dax-udf-functions-guidelines.md` — advanced section):** `(param [: Type [Subtype] [Val|Expr]], …) => body`; Val (eval at call site) vs **Expr** (substituted/re-evaluated inner — needed for TI, e.g. `PriorYearValue(expr: Scalar Variant Expr, dateColumn: AnyRef) => CALCULATE(expr, SAMEPERIODLASTYEAR(dateColumn))`).
- **Correctness gotcha (dg3 refresh notes):** `SUMMARIZECOLUMNS` inside `CALCULATETABLE` can introduce circular deps (Sept-2024 engine change) — add the grouped tables as explicit filters inside SUMMARIZECOLUMNS.
**Mix-and-match notes:** DAX001–021 + engine model + valid-query rules from **powerbi-agentic** (deepest); advanced templates from **awesome-copilot + powerbi-master** (used only to ensure coverage breadth, prefer first-party phrasing on overlap); UDF section from **powerbi-agentic**.
**Reference-only / defer:** all trace-metric derivations (TotalDuration/SE Parallelism Factor etc.) — only actionable with live-trace tooling we don't have; xmSQL/`$SYSTEM` DMV mechanics; the data-goblin Decision-Guide trace-only rows (`__ValueFilterDM`, CallbackDataID signals).
**Conflicts resolved:** **DIVIDE conflict (see Known Conflict #1 below).** Stated rule for the skill: *"Use `DIVIDE()` by default for measure-level division (handles divide-by-zero). EXCEPTION (DAX018): inside a row-context iterator (SUMX/AVERAGEX/etc.) where the denominator is guaranteed non-zero, plain `/` is faster because DIVIDE forces an FE callback per row; if not guaranteed, pre-filter `<>0`."* model-reviewer must NOT flag `/` inside an iterator with a guarded denominator.
**Provenance:** `powerbi-agentic-plugins.md` (DAX framework + query guide + UDF), `dg3-semantic-models.md` (DAX001–021 verbatim + Decision Guide + circular-dep gotcha), `awesome-copilot-pbi-data.md` (advanced templates), `claude-plugin-marketplace-pbi.md` (advanced DAX + anti-pattern table).

## time-intelligence
**Verdict:** rich content available.
**Canonical source(s):** dg3/powerbi-agentic for the **date-table-correctness rule + TI-function detector + lift-TI-to-outer-CALCULATE perf rules**; dg4 + powerbi-master + skills-for-fabric for **calc-group TI item bodies** (shared with calc-group-patterns); dg2 for **target/comparison DAX templates**.
**Adopt (merged):**
- **Date-table correctness rule (dg3 — verbatim-worthy, the load-bearing one):** a date table MUST be marked `dataCategory: Time`, have a key Date column, **continuous daily dates (no gaps)**, span the full fact range, and relate via a single-column relationship. Missing any → DATEADD/SAMEPERIODLASTYEAR/TOTALYTD/DATESYTD return BLANK. Prefer a source date table; disable Auto Date/Time; `sortByColumn` month-name → month-number.
- **Canonical TI-function detector (powerbi-agentic — reuse across this skill, the DQ-TI BPA rule, and DAX019/020 detection):** the ~35-function regex list from `MEASURES_USING_TIME_INTELLIGENCE_AND_MODEL_IS_USING_DIRECT_QUERY` (DATESYTD/MTD/QTD, TOTALYTD/MTD/QTD, SAMEPERIODLASTYEAR, DATEADD, DATESINPERIOD, DATESBETWEEN, PARALLELPERIOD, PREVIOUS/NEXT-DAY/MONTH/QUARTER/YEAR, FIRSTDATE/LASTDATE, OPENING/CLOSINGBALANCE*, etc.).
- **TI off-by-one + anchor rules (powerbi-agentic `dax-query-guidelines.md`):** DATESINPERIOD offset must match the window exactly (12-month = −12, not −11); determine "last year" from `MAX('Sales'[OrderDate])` not the Calendar max (avoids trailing empty dates); establish date context via a date groupby column or date filter.
- **Performance rules (cross-listed from dax-patterns):** DAX019 keep base measures TI-free, apply TI once in an outer CALCULATE (`CALCULATE([Revenue]-[Cost], DATESYTD('Date'[Date]))`); DAX020 lift TI/dynamic filters out of per-slice measures; MDL005 pre-compute period-comparison columns (store `SalesLY` instead of SAMEPERIODLASTYEAR — one scan); MDL006 row-based TI table so all period measures fuse.
- **Target / comparison templates (dg2 — for kpi-design-rules cross-reference, all generic):** prior-year `CALCULATE([m], DATEADD('Date'[Date],-1,YEAR))`; prior-period `…,-1,MONTH`; rolling avg `CALCULATE([m], DATESINPERIOD('Date'[Date], MAX('Date'[Date]), -3, MONTH))`; 3-month moving avg via `EDATE(MAX,-2)`+`DATESBETWEEN`; QoQ via `DATEADD(…,-1,QUARTER)`; fiscal-YTD pattern.
**Mix-and-match notes:** correctness rule + detector + perf rules from **dg3/powerbi-agentic**; calc-group item bodies shared verbatim with **calc-group-patterns** (single canonical copy); target templates from **dg2**.
**Reference-only / defer:** week-based time-intelligence (powerbi-master) — niche, keep as reference.
**Conflicts resolved:** none specific (DIVIDE handled in dax-patterns).
**Provenance:** `dg3-semantic-models.md` (date-table correctness, MDL005/006), `powerbi-agentic-plugins.md` (TI detector, off-by-one), `dg2-reports.md` (target DAX), `dg4-te-fabric-hooks-root.md` + `claude-plugin-marketplace-pbi.md` (calc-group TI bodies).

## m-query-patterns
**Verdict:** rich content available.
**Canonical source(s):** **dg3 `power-query/best-practices.md` = primary** (the most complete query-folding catalog in the corpus). skills-for-fabric `DATAFLOWS-AUTHORING-CORE` adds **section-document structure + connection-kind map**; powerbi-master + awesome-copilot add **recipes** (calendar generator, REST pagination, incremental-refresh M).
**Adopt (merged):**
- **Safe write order (maximizes folding):** 1) filter rows → WHERE, 2) select columns → SELECT, 3) set types → CAST, 4) sort → ORDER BY, 5) non-foldable transforms LAST. Golden rule: "all foldable work first, then non-foldable." Native escape hatch: `Value.NativeQuery(Source,"SELECT …",null,[EnableFolding=true])`.
- **Folds → SQL (adopt the catalog):** `Table.SelectColumns/RemoveColumns`(SELECT), `Table.SelectRows`(WHERE), `Table.Sort`(ORDER BY), `Table.FirstN`(TOP), `Table.Group`(GROUP BY), `Table.TransformColumnTypes`(CAST), `Table.RenameColumns`(AS), `Table.ExpandTableColumn`/`Table.NestedJoin`(JOIN), `Table.Distinct`(DISTINCT), `Table.Skip`(OFFSET).
- **Breaks folding (the full list — adopt as the lint signal):** `Table.Buffer`/`List.Buffer`/`Table.StopFolding`; `#table`/`Table.FromList/FromRecords/FromRows/FromColumns`; index/position ops (`Table.AddIndexColumn`, `Table.LastN`, `Table.Range`, `Table.ReverseRows`); most `Text.*` (Proper/Combine/Split/Pad/Reverse/Format); `Table.SplitColumn`/`CombineColumns`; `Table.Transpose`/`Promote/DemoteHeaders`; `Table.FillDown/FillUp`; error handling (`Table.RemoveRowsWithErrors`, `try…otherwise` in row context); schema/metadata (`Table.Schema`, `Table.ColumnNames`); custom `(x)=>` lambdas, `Table.TransformRows`, `List.Generate/Accumulate`; locale date-to-text & relative-date funcs. Plus environmental fold-breakers: merged different sources, incompatible privacy levels (firewall), flat-file/Web.Contents sources, custom SQL without EnableFolding, any step after a fold-breaker.
- **Sometimes-folds:** `Table.AddColumn`/`TransformColumns` only with SQL-translatable funcs (Text.Upper/Lower/Trim, Number.Round); `Table.Pivot/Unpivot` (SQL Server only); `Table.SelectRows`+`Text.Contains`(→LIKE); `Date.Year/Month/Day`(→YEAR/MONTH/DAY); `Date.AddDays/AddMonths`(→DATEADD); `Text.Start/End`(→LEFT/RIGHT).
- **Type mappings + naming + error handling:** `Int64.Type`, `type text/date/datetime/datetimezone/logical`, `Currency.Type`, `Percentage.Type`; descriptive `#"Quoted Step Names"`, PascalCase parameters; **avoid silent `try…otherwise` in production partitions** (a failed refresh beats silently wrong data).
- **Section-document structure (skills-for-fabric):** `section Section1;` → `shared QueryName = let … in …;`; parameters `shared ServerName = "…" meta [IsParameterQuery=true, Type="Text"];`; connection-kind ↔ M-function map (`Lakehouse`→`Lakehouse.Contents()`, `Sql`→`Sql.Database()`, `Web`→`Web.Contents()`, `SharePoint`→`SharePoint.Contents()`, `AzureDataLakeStorage`→`AzureStorage.DataLake()`); query classification heuristic (`loadEnabled`→OUTPUT, `isHidden`→HELPER, else STAGING).
- **Incremental-refresh M (awesome-copilot — adopt the folding pattern):** filter on `RangeStart`/`RangeEnd` + `Int32.From(DateTime.ToText(…,"yyyyMMdd"))`; note `Value.NativeQuery(…,[EnableFolding=false])` disables folding.
- **Calendar-table generator (powerbi-master — reusable date-dim scaffold):** `List.Dates` + Year/Month/Quarter/Weekday/FiscalYear/IsWeekend.
- **PQ anti-pattern rules (dg4 — feed BPA M-scope rules):** flag `Table.Distinct`, `Table.RemoveDuplicates`, `Table.NestedJoin`/`Table.Join`, `Table.FuzzyJoin`/`FuzzyNestedJoin`, `Table.Combine`+distinct, missing buffer-before-join; canonical MS rule `MINIMIZE_POWER_QUERY_TRANSFORMATIONS`.
- **Layering discipline (antigravity dbt playbook — methodology):** staging (1:1 source, light cleaning, rename) → intermediate (business logic/joins) → marts (final). Mirror as raw-fold layer vs business layer.
**Mix-and-match notes:** folding catalog from **dg3** (deepest); section-document + connection map from **skills-for-fabric**; recipes (calendar/pagination/incremental) from **powerbi-master + awesome-copilot**; layering methodology from **antigravity**.
**Reference-only / defer:** M *validation* via `executeQuery`/XMLA save (dg3 `validation.md`) — concept is useful ("XMLA save validates M syntax but misses column/connectivity/folding errors") but the Fabric/Python execution is non-Node; SCD-Type-1 hash-change M (awesome-copilot) — advanced, keep as reference.
**Conflicts resolved:** none — sources are complementary (dg3 is folding-deep, others are recipe-deep).
**Provenance:** `dg3-semantic-models.md` (§6 folding catalog, §7 validation), `skills-for-fabric-authoring.md` (§ Power Query M / Dataflows), `claude-plugin-marketplace-pbi.md` (M cookbook), `awesome-copilot-pbi-data.md` (incremental refresh M), `antigravity-bi-bundles.md` (dbt layering).

## rls-patterns
**Verdict:** rich content available.
**Canonical source(s):** **awesome-copilot `RLS.md` = best-organized** (static vs dynamic, four patterns, common-mistakes, OLS, checklist). dg4 + skills-for-fabric add the **TMDL role-file syntax**; awesome-copilot `…security-rls` instructions add **CUSTOMDATA/time-based/least-privilege**.
**Adopt (merged):**
- **Principles:** **filter dimensions not facts** (smaller tables + propagation); minimal roles (each role = a separate cache); **roles are additive/UNION, not intersection**; prefer dynamic RLS via `USERPRINCIPALNAME()`; admins/members bypass RLS; `USERPRINCIPALNAME()` ≠ email; bidirectional + RLS can fail silently; test in the Service.
- **Filter-expression library (all `'Table'[Col]` are placeholders):**
  - Static: `'Region'[Region] = "West"`; multi-value `IN { "West", "Central" }`.
  - Dynamic identity: `'Employee'[Email] = USERPRINCIPALNAME()`.
  - **Security-table (the recommended scalable pattern):** `'Sales'[RegionID] IN CALCULATETABLE(VALUES('UserRegions'[RegionID]), 'UserRegions'[Email] = USERPRINCIPALNAME())`.
  - Manager hierarchy: `PATHCONTAINS('Employee'[ManagerPath], USERPRINCIPALNAME())`.
  - **Defensive default-deny (adopt as the recommended template):** `IF(USERPRINCIPALNAME() IN VALUES(SecurityMapping[UserEmail]), [Region] IN SELECTCOLUMNS(FILTER(SecurityMapping,[UserEmail]=USERPRINCIPALNAME()),"Region",[Region]), FALSE())`.
  - `CUSTOMDATA()` (embedded scenarios) with a SWITCH + `FALSE()` default; time-based via a per-role cutoff date.
- **TMDL role-file syntax (dg4 + skills-for-fabric):** `role <Name>` → `modelPermission: read` (or `readRefresh`) → `tablePermission <Table> = <DAX filter>` (one per table) → add `ref role` in model.tmdl. **Role membership is NOT in TMDL** (assigned via REST/Entra after deploy). OLS = `columnPermission Sales.'Unit Cost' = none` / hide a table via `MetadataPermission = None`.
- **Common mistakes (adopt as reviewer checks):** RLS on fact tables only (perf); LOOKUPVALUE instead of relationships; expecting intersection; forgetting DirectQuery (filters → WHERE); overly-permissive `TRUE()` default; not testing valid/invalid/NULL users.
- **Cross-link to BPA:** rules `LIMIT_ROW_LEVEL_SECURITY_(RLS)_LOGIC` (no RIGHT/LEFT/UPPER/LOWER/FIND), `CHECK_IF_DYNAMIC_RLS_IS_NECESSARY` (USERNAME/USERPRINCIPALNAME), `AVOID_USING_MANY-TO-MANY…DYNAMIC_RLS`, `AVOID_THE_USERELATIONSHIP_FUNCTION_AND_RLS_AGAINST_THE_SAME_TABLE`.
**Mix-and-match notes:** principles + pattern library from **awesome-copilot RLS.md**; TMDL emission syntax from **dg4 + skills-for-fabric**; advanced (CUSTOMDATA/time/least-privilege) from **awesome-copilot instructions**.
