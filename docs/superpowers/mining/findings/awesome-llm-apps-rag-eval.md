# Mining findings: awesome-llm-apps — grounding/memory/eval patterns
Source: awesome-llm-apps.xml

## Relevance summary
Signal-to-noise is moderate. The repo is ~100 demo apps and the pack is README-only, so most entries are "pip install + streamlit run" tutorials with no extractable engineering depth. BUT a focused minority is genuinely high-value for us: the RAG tutorials encode reusable grounding/citation/relevance-grading patterns; one optimization tool and one OpenAI-SDK lesson give us context-discipline and validation-gate vocabulary; and a small cluster (code-reviewer skill, fact-checker skill, self-improving-skills, critique-loop, agent-governance) is a near-perfect blueprint for our reviewer agents, anti-fabrication gates, and eval harness. Memory tutorials are the weakest area — five near-identical Mem0+Qdrant clones, so the pattern is worth one extraction, not five. I discarded the bulk (game-playing, finance/legal/travel agent teams, finetuning, voice) as app-specific noise.

## High-value extractions

### Grounded generation: same retrieval packet feeds both answer AND citations → maps to our DAX-grounding gate / no-fabrication invariant
- The pattern (runtime-agnostic): Retrieval runs exactly once; the *identical* ranked-evidence packet is passed both to the answer-generation step and to the citation panel. The model literally cannot cite (or assert) anything that was not in the retrieved set. Citations are rendered as a separate structured list keyed by ID, not inlined free-text.
- Why it helps / how to apply: This is the cleanest anti-fabrication invariant in the whole source and it generalizes directly. For us: when an agent authors a measure/visual, the *same* resolved model index (the exact set of tables/columns/measures we read from TMDL) that we feed to the generation prompt is the *only* set the validator accepts references from. One source of truth, retrieved once, used for both "write" and "verify" — the model can't reference a column it wasn't shown. Render the "fields used" as a structured list the gate checks, not prose.
- Source path within repo: `rag_tutorials/multimodal_agentic_rag/README.md` (also reinforced by `rag_tutorials/knowledge_graph_rag_citations/README.md`)
- Quality: 5 — concise, specific, load-bearing invariant.
- Recommendation: adapt

### Provenance-tracked knowledge graph with verifiable citations + multi-hop traversal → maps to our semantic-model graph + DAX-reference check
- The pattern (runtime-agnostic): Build a structured graph of entities + typed relationships from the source (not just text chunks). Every extracted node/edge carries provenance (source doc + chunk). Answering = traverse relationships (multi-hop) to assemble context, then every claim links back to a specific source entry. Output separates Answer / Citations (source + exact text) / Reasoning-path.
- Why it helps / how to apply: A Power BI semantic model *is* a graph (tables → columns/measures, relationships, measure→measure/column dependency edges). This validates building an in-memory model graph and resolving every DAX reference by graph traversal rather than fuzzy text match — exactly our "verify DAX refs against the model" gate. The "reasoning-path + source-text per claim" output is a great shape for a reviewer agent's report: for each referenced field, emit where it exists in the model. Vector/text-similarity RAG explicitly "struggles to provide verifiable sources" — graph traversal gives exact provenance, which is what we want for correctness, not similarity.
- Source path within repo: `rag_tutorials/knowledge_graph_rag_citations/README.md`
- Quality: 5 — directly analogous to our domain; clear on why graph > vector for verifiability.
- Recommendation: adapt

### Corrective RAG: relevance-grade → transform query → fallback (don't answer on weak evidence) → maps to reviewer/grading gate before action
- The pattern (runtime-agnostic): A multi-stage corrective loop: (1) retrieve, (2) an LLM *grades each retrieved doc for relevance*, (3) if evidence is insufficient, *transform/rewrite the query* and/or fall back to another source rather than answering from weak context. Echoed by `corrective_rag`, `ai_blog_search` (relevance grading + query refinement + "rewrite vs answer vs retrieve-more" decision), `deepseek_local_rag_agent` and `rag_database_routing` (similarity threshold filtering, then web fallback).
- Why it helps / how to apply: The reusable idea is a *grading gate that can refuse*: don't let the agent proceed on insufficient grounding. For us: before authoring DAX, if the requested fields/semantics can't be confidently resolved against the model, the agent should rewrite/clarify the request or report "cannot ground" — never fabricate a plausible column. The "grade-then-decide (answer / refine / refuse)" control flow is a reviewer-agent skeleton.
- Source path within repo: `rag_tutorials/corrective_rag/README.md`, `rag_tutorials/ai_blog_search/README.md`, `rag_tutorials/rag_database_routing/README.md`
- Quality: 4 — strong control-flow pattern; READMEs describe it but don't show the grader prompt.
- Recommendation: adapt

### RAG failure-pattern taxonomy (P01–P12) → maps to reviewer-agent triage vocabulary + our test-fixture catalog
- The pattern (runtime-agnostic): A small, fixed library of ~12 named failure patterns (e.g., P01 grounding drift / answer contradicts retrieved docs; P05 router misalignment; P06 long-chain reasoning drift loses earlier constraints; P07 ungrounded tool calls / wrong args; P09 eval blind spots — passes tests, fails real incidents; P12 multi-agent interference). An incident is classified to one primary pattern (+ optional secondaries) and the system proposes a *minimal structural fix*, not "add more context / better model." Diagnoses are appended to a JSON incident library.
- Why it helps / how to apply: This is a ready-made vocabulary for our reviewer subagents and for naming our anti-fabrication test fixtures. P01/P07 are exactly the failure we gate against (DAX referencing non-existent fields = grounding drift / ungrounded tool call). P09 is a direct warning for our eval design: green unit tests ≠ no hallucination on real models. Adopt the "classify to a named pattern → propose minimal structural fix" framing for review output, and keep a small JSON library of real grounding-failure cases as regression fixtures.
- Source path within repo: `rag_tutorials/rag_failure_diagnostics_clinic/README.md`
- Quality: 5 — the most directly reusable artifact in the source; framework-agnostic by design.
- Recommendation: adopt (the taxonomy + fix-framing); reference-only for the script itself

### Reviewer agent = severity-ranked rule library with examples + structured verdict → maps to our BPA-style reviewer + DAX gate output
- The pattern (runtime-agnostic): A code-reviewer skill where each rule is a record: name + impact (CRITICAL/HIGH/MEDIUM/LOW) + category + tags + ❌incorrect/✅correct examples + link to a detailed rule file. The skill enforces a *priority order* (Security → Performance → Correctness → Maintainability), provides a checklist, a severity→action table (CRITICAL = block merge), and a fixed output format (issues grouped by severity, each with file:line + issue + fix, plus a counts summary and an overall recommendation). The thin SKILL.md points to a `rules/` directory (one file per rule) — detail loaded on demand.
- Why it helps / how to apply: This is our reviewer/BPA agent design, almost verbatim. Define our checks (e.g., "DAX references unknown column" = CRITICAL/blocking; "measure missing format string" = MEDIUM) as a rule library with severity + good/bad examples, enforce a severity→action gate (CRITICAL blocks the write), and emit a structured verdict with counts. The "SKILL.md is an index; rules live in separate files loaded on demand" is also a context-discipline win (progressive disclosure — don't dump every rule into context). Rules must be authored dataset-agnostically (about model *structure*, never specific field names).
- Source path within repo: `awesome_agent_skills/code-reviewer/AGENTS.md` + `awesome_agent_skills/code-reviewer/SKILL.md`
- Quality: 5 — clean, complete reviewer blueprint with severity gating and output schema.
- Recommendation: adapt

### LLM-as-judge: systematic verification with a rating scale, evidence requirement, and source hierarchy → maps to a grounding/quality reviewer agent
- The pattern (runtime-agnostic): A fact-checker skill with a fixed verification process: (1) isolate the exact claim, (2) state what evidence would prove/disprove it, (3) evaluate available evidence, (4) assign a *discrete rating* from a fixed scale (TRUE / MOSTLY-TRUE / MIXED / MOSTLY-FALSE / FALSE / UNVERIFIABLE), (5) add context. Includes a source-credibility hierarchy and a strict output template (Claim / Verdict / Analysis / Evidence / Sources). Key move: separate verifiable fact from opinion, and have an explicit "UNVERIFIABLE" verdict.
- Why it helps / how to apply: This is the judge half of an eval harness. For grading agent output, define a discrete rubric (e.g., GROUNDED / PARTIALLY-GROUNDED / UNGROUNDED / UN-CHECKABLE) and require the judge to cite the model evidence for each verdict. The "UNVERIFIABLE/UN-CHECKABLE" escape hatch is important — it prevents the judge from itself fabricating a pass/fail when the model can't be resolved. The "state required evidence before judging" step forces grounding. Fits both an interactive reviewer subagent and an automated eval scorer.
- Source path within repo: `awesome_agent_skills/fact-checker/SKILL.md`
- Quality: 5 — disciplined, reusable judge structure with a discrete scale + evidence step.
- Recommendation: adapt

### Eval harness: Executor/Analyst/Mutator loop with binary criteria + keep-if-improved → maps to our test-fixture eval + reviewer-driven iteration
- The pattern (runtime-agnostic): A three-role optimization loop. Executor runs the artifact against a set of test scenarios and scores each output against *binary yes/no eval criteria* (establishing a baseline). Analyst diagnoses failures, identifies root cause, and selects a *named mutation strategy* (add_example / add_constraint / restructure / add_edge_case) — returning *structured JSON via output_schema*. Mutator applies exactly ONE surgical change. Re-run, re-score, *keep the change only if score improved, else revert*. Bounded by target pass-rate and max_rounds.
- Why it helps / how to apply: This is a complete, runtime-agnostic eval-harness blueprint we can mirror in Node/Claude Code. Concrete adoptions: (a) score agent output against *binary yes/no criteria* (e.g., "every DAX reference resolves in the model? y/n", "measure has a format string? y/n") — binary is more reliable than fuzzy scores; (b) force the diagnosing/judging agent to emit *structured output* (a schema), not prose, so results are machine-checkable; (c) apply ONE change at a time and keep-if-improved-else-revert to avoid regressions; (d) bound iterations (max_rounds) to prevent runaway loops. Use it both to evaluate generated artifacts and to harden our skills/prompts against fixtures.
- Source path within repo: `awesome_agent_skills/self-improving-agent-skills/README.md`
- Quality: 5 — the most complete eval/iteration loop in the source; the binary-criteria + structured-output + revert details are load-bearing.
- Recommendation: adapt

### Generate → critique → revise loop (self-refinement with a paper trail) → maps to author-then-review before commit
- The pattern (runtime-agnostic): Bounded iterative refinement: generate an initial answer (here via parallel candidates synthesized at low temperature), then a *critic* identifies missing info / unclear / logical flaws, then a *reviser* fixes every critique point while preserving good parts; repeat for N (1–3) rounds. The critique and revision at each step are kept as a transparent paper trail.
- Why it helps / how to apply: A lightweight reviewer pass that doesn't need a separate eval dataset — good for in-session quality before we write to disk. For us: author the DAX/visual, run a critic pass (does every reference resolve? is the measure logically what was asked? naming/format conventions?), revise, cap at a small number of rounds, and surface the critique trail so the user sees *why* it changed. Pairs with the binary-criteria harness above (this is the in-loop refiner; that is the offline scorer).
- Source path within repo: `advanced_llm_apps/gpt_oss_critique_improvement_loop/README.md`
- Quality: 4 — solid, simple pattern; bounded-rounds + paper-trail are the keepers.
- Recommendation: adapt

### Deterministic policy gate: intercept action BEFORE execution, decide allow/deny/approve → maps directly to our pre-write validation hook
- The pattern (runtime-agnostic): A governance layer that wraps tools and intercepts every action *before* it executes. A *deterministic* policy engine (declarative YAML, not LLM judgment) evaluates the action and returns ALLOW / DENY / REQUIRE-HUMAN-APPROVAL, with everything logged to an audit trail. Policies cover filesystem scope, allowlists, rate limits, and "require approval for" specific actions.
- Why it helps / how to apply: This is the strongest validation-gate pattern for reliability because the gate is *deterministic code, not a model*. It validates our architecture: a pre-write hook intercepts "create/modify measure" and runs a deterministic DAX-reference check against the model index → ALLOW only if every reference resolves; otherwise DENY (or flag for the user). Don't rely on the authoring LLM to police itself — interpose a code gate. The ALLOW/DENY/REQUIRE-APPROVAL trichotomy is a good fit (auto-allow clean writes, deny ungrounded ones, ask the human on ambiguous ones). Declarative policy config keeps it dataset-agnostic.
- Source path within repo: `advanced_ai_agents/single_agent_apps/ai_agent_governance/README.md`
- Quality: 5 — deterministic-gate-before-execution is exactly our hook model.
- Recommendation: adapt

### Layered input/output guardrails with tripwires + confidence thresholds → maps to pre- and post-write validation around the agent
- The pattern (runtime-agnostic): OpenAI-SDK guardrails: *input guardrails* validate/transform before the agent runs; *output guardrails* validate before delivery; either can fire a *tripwire* that hard-blocks with a typed exception. Validation can be delegated to a specialized *guardrail agent* with structured output. Best practices: layered defense (multiple guardrails), *confidence-threshold blocking* to reduce false positives, clear block messages, and logging every decision. Related: the agent loop bounds iterations with `max_turns`/`MaxTurnsExceeded`.
- Why it helps / how to apply: Frames our gates as a symmetric pair: an *input* guardrail (is the request groundable against the model? is required context present?) and an *output* guardrail (does the produced TMDL/DAX pass the reference check?) with a tripwire that blocks the write. The confidence-threshold idea matters for any LLM-based check we add — block only above a threshold to avoid over-blocking valid authoring. Bounding agent turns (max_turns) is a simple safety rail against runaway loops.
- Source path within repo: `ai_agent_framework_crash_course/openai_sdk_crash_course/6_guardrails_validation/README.md` (+ `4_running_agents/README.md` for max_turns)
- Quality: 4 — clear input/output/tripwire framing; confidence-threshold and layering are the keepers.
- Recommendation: adapt

### Typed, read-only shared context object passed to every tool → maps to passing the model index as typed context
- The pattern (runtime-agnostic): A `RunContextWrapper`-style typed context object holds shared state (user/session/external-system handles) and is passed to the runner so *every* tool accesses the same context with compile-time type safety. Best practices: use frozen/immutable context for read-only data, validate on construction, document fields.
- Why it helps / how to apply: Our resolved model index (tables/columns/measures/relationships) should be exactly this: a typed, validated, *immutable* context object threaded to every tool and to the validation gate — so generation, validation, and review all read one consistent snapshot (reinforces the "retrieved once, used everywhere" invariant above). TypeScript makes this natural. Immutability prevents an agent from mutating the ground truth mid-run.
- Source path within repo: `ai_agent_framework_crash_course/openai_sdk_crash_course/5_context_management/README.md`
- Quality: 4 — good context-discipline framing; immutability + single-snapshot are the keepers.
- Recommendation: adapt

### Statistical tool-output compression with safety invariants → maps to compacting model/TMDL reads before they hit context
- The pattern (runtime-agnostic): "Headroom" compresses redundant tool outputs (logs, search results, large JSON) by keeping first items + last items + anomalies + query-relevant matches, with reversible "compress-cache-retrieve" so the model can pull the original on demand. Hard safety invariants: never drop human/user content; never break tool-call/response pairing; parse failures are no-ops (pass through unchanged). Includes prefix stabilization for provider prompt-cache hits.
- Why it helps / how to apply: Large semantic models produce huge TMDL/field dumps that blow context. The reusable discipline: feed the agent a *query-relevant slice* of the model (keep what matters: the tables/fields relevant to the task + anomalies) rather than the whole model, but keep the full index retrievable on demand so grounding stays exact. Crucially adopt the safety invariants: never silently corrupt the source, and if you can't parse/slice, pass through unchanged rather than guess. Stable prompt prefixes → better cache hits / lower cost.
- Source path within repo: `advanced_llm_apps/llm_optimization_tools/headroom_context_optimization/README.md`
- Quality: 4 — concept + safety invariants are reusable; specific library is Python-only.
- Recommendation: reference-only (concept + invariants); adapt the slicing idea

### TOON: compact tabular serialization for LLM context → maps to how we serialize field lists / model metadata to the agent
- The pattern (runtime-agnostic): TOON (Token-Oriented Object Notation) serializes uniform tabular data far more compactly than JSON (header row once, then CSV-like rows), reportedly ~40–60% token reduction for tabular/uniform data, while staying human-readable and explicitly structured. Use for uniform tables; keep JSON for irregular/nested data.
- Why it helps / how to apply: Most of what we hand an agent is *uniform tabular metadata* — column lists, measure inventories, relationship tables. Serializing those as a header-once tabular block instead of repeated-key JSON cuts tokens substantially with no loss of grounding fidelity, leaving more budget for reasoning. Pure context-discipline, dataset-agnostic. Evaluate before adopting — token savings depend on uniformity, and irregular metadata should stay JSON.
- Source path within repo: `advanced_llm_apps/llm_optimization_tools/toonify_token_optimization/README.md`
- Quality: 3 — useful, low-risk context optimization, but a marketing-flavored README; benchmark on our own metadata first.
- Recommendation: reference-only (consider adopting the encoding for field/metadata lists)

### Persistent agent memory via external store (Mem0 + vector DB) → maps to lightweight cross-session project memory (low priority for us)
- The pattern (runtime-agnostic): Persist user/session facts and past interactions in an external memory layer (Mem0 + a vector store), retrieved to personalize later turns; memory is keyed per user/session and survives restarts. The same pattern recurs ~5× across the memory tutorials with only the model swapped.
- Why it helps / how to apply: Mostly *low* relevance for our anti-fabrication goal — conversational personalization isn't our problem. The transferable nugget: keep durable project context (e.g., naming conventions, prior decisions, recurring model quirks) in an *external* store keyed to the project, retrieved into context when relevant — and keep it *separate* from the ground-truth model index (memory is preference/history; the model is truth — never let memory override the model in grounding). The trust-gated app reinforces "store the durable record externally, not in the agent's own memory."
- Source path within repo: `advanced_llm_apps/llm_apps_with_memory_tutorials/*` (5 near-duplicates: `ai_arxiv_agent_memory`, `ai_travel_agent_memory`, `llm_app_personalized_memory`, `local_chatgpt_with_memory`, `multi_llm_memory`)
- Quality: 2 — one idea repeated five times; thin on engineering detail; tangential to grounding.
- Recommendation: reference-only

Sibling-agent orchestration findings will likely also touch the Executor/Analyst/Mutator multi-role loop.
- Deterministic pre-execution gate (`ai_agent_governance`) + input/output guardrails (`openai_sdk`) describe the same "validate before the write commits" idea our project already implements as a hook — cross-reference with the hooks/validator findings from the Claude-plugin and fabric sources so we don't design three competing gate mechanisms.
- "Retrieve once, use the same packet for answer + citations" (`multimodal_agentic_rag`) and "typed immutable shared context" (`openai_sdk` context mgmt) and "graph provenance" (`knowledge_graph_rag_citations`) are three angles on ONE invariant: a single immutable model snapshot is the sole grounding source — consolidate into one principle in the consolidated findings.
- Memory-via-vector-store overlaps with any memory findings from other sources; flag the five duplicates here as a single pattern to avoid double-counting.
