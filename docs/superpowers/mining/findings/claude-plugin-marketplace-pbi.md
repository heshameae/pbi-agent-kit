# Mining findings: claude-plugin-marketplace — powerbi-master + data plugins
Source: claude-plugin-marketplace.xml

## Relevance summary
`powerbi-master` (Josiah Siegel, v0.5.1, "2026"-dated) is a genuinely strong, well-structured PBI plugin: 10 skills + 1 agent + 2 commands, all aligned with the same modern stack we target (TMDL GA, PBIR enhanced format, PBIP, BPA, fabric-cicd). Its highest-value, directly-reusable assets are the **TMDL syntax/grammar reference** (TMDL→TOM 1:1 keyword/property map), the **TMDL examples cookbook** (calc groups, RLS/OLS, field params, KPIs, incremental refresh), the **four-layer validation model** (parser → TOM schema → BPA → lineage) with a concrete **PBIR lineage-linter** and **BPA rule catalog**, and the **PBIR JSON schema reference** (visual.json/page.json query buckets + formatting `expr.Literal.Value` object model). The data plugins (`tsql-master`, `ssdt-master`, `adf-master`) are T-SQL/PowerShell/Azure-runtime and are reference-only per our Node/TS rule — a few SQL patterns (window functions, recursive-CTE hierarchies, GROUPING SETS, MERGE/SCD) are conceptual parallels to DAX/M but contain no transferable code. Note: powerbi-master is broad and accurate but, like other 3rd-party PBI sources, somewhat thinner on report-authoring depth than the first-party data-goblin PBIP/PBIR repos; its TMDL grammar table and validation taxonomy are where it adds the most net-new value.

## High-value extractions

### TMDL → TOM 1:1 grammar & property reference → maps to tmdl-conventions + `modeling/tmdl-parser.ts` + `modeling/types.ts`
- What it is / why valuable: The single most authoritative TMDL grammar doc in the pack. Maps every TMDL keyword to its TOM class, lists the *default property* per object type, the expression-language per property, indentation rules, ref/createOrReplace semantics, reserved keywords, and quoting/escaping rules. This is exactly the spec our `tmdl-parser` must honor and our `types.ts` should model.
- Key content (reusable bits):
  - Top-level objects (no indent): `database, model, table, relationship, role, culture, perspective, expression, function (DAX UDF), dataSource`.
  - Table children: `column` (DataColumn), `column =` (CalculatedColumn, default prop = Expression/DAX), `measure` (default = Expression/DAX), `partition` (default = SourceType; `= m`/`= calculated`/`= entity`/`= query`), `hierarchy`, `calculationGroup`, `calculationItem`, `annotation` (default = Value/Text).
  - Default-property + language table: measure/calcColumn/calcItem → DAX; partition(m)/namedExpression → M; partition(calculated)/tablePermission/formatStringDefinition/detailRowsDefinition/KPI exprs → DAX; annotation → Text; jsonExtendedProperty → JSON.
  - Property delimiters: `:` for scalar/reference props, `=` for the default expression; boolean shortcut (`isHidden` ⇒ true); `///` lines above an object become its TOM `Description`.
  - Names quoted in single quotes if they contain dot/equals/colon/quote/whitespace; escape internal `'` by doubling. Object names matching reserved keywords MUST be quoted.
  - Indentation: strict, **single tab per level** (mixed tabs/spaces → `TmdlFormatException`). Multi-line expressions indent one level deeper than parent props. Triple-backtick blocks preserve exact whitespace.
  - `ref` keyword defines deterministic collection ordering in model.tmdl; objects referenced-but-missing are ignored; objects present-but-unref'd are appended. `createOrReplace` = one verb per script, replaces object + all descendants.
  - Full per-object property tables (column: dataType enum, summarizeBy enum, sourceColumn, sortByColumn, displayFolder, dataCategory, isKey/isHidden/isNullable/isUnique/isAvailableInMdx; relationship: crossFilteringBehavior/securityFilteringBehavior enums, fromCardinality/toCardinality, isActive, relyOnReferentialIntegrity, joinOnDateBehavior; role: modelPermission enum, tablePermission DAX, columnPermission, member types user/group/auto/activeDirectory).
- Source path: `plugins/powerbi-master/skills/tmdl-mastery/references/tmdl-syntax-reference.md` (also SKILL.md at `skills/tmdl-mastery/SKILL.md` for the casing/whitespace/partial-declaration rules and TMDL-vs-TMSL-vs-BIM matrix).
- Quality: 5 — precise, complete, matches the real TMDL spec.
- Recommendation: adopt-as-is (as a knowledge doc for tmdl-conventions; cross-check against our parser's enum handling and quoting/escaping). Verify our parser handles `///` descriptions, triple-backtick blocks, and partial declarations.

### TMDL examples cookbook (every object type, copy-pasteable) → maps to tmdl-conventions, calc-group-patterns, rls-patterns, time-intelligence, kpi/table design
- What it is / why valuable: Correctly-indented, end-to-end TMDL for database/model/relationships/expressions/functions, a fully-fleshed fact table (partitions, key/date/fact/calculated columns, measures, dynamic format string, detailRowsDefinition, annotations), a date-dim with hierarchy, a dimension with display folders, **two calculation groups** (time-intelligence + currency with `formatStringDefinition`), **field parameters** (with `ParameterMetadata` annotation), **RLS roles** (incl. group/auto members), **OLS** (columnPermission none), perspectives, cultures/translations, KPI measure block, DirectQuery/Dual partitions, and a complete **incremental refresh `refreshPolicy`** (granularity/periods/pollingExpression/sourceExpression). These are ready templates for our scaffold/measure/calc-group skills.
- Key content (highest-value snippets):
  - Time-intelligence calc group items: `Current = SELECTEDMEASURE()`; `YTD = CALCULATE(SELECTEDMEASURE(), DATESYTD('Calendar'[Date]))`; `QTD/MTD` analogues; `PY = CALCULATE(SELECTEDMEASURE(), SAMEPERIODLASTYEAR('Calendar'[Date]))`; `'YoY %'` with `formatStringDefinition = "0.00%"` and `ordinal`. Requires `column 'Name'` + hidden `Ordinal` sort column — a pattern our calc-group scaffolding must emit.
  - Field parameter calculated-table partition shape: `{ ("Revenue", NAMEOF([Sales Amount]), 0), ... }` + required `ParameterMetadata` annotation `{"version":3,"kind":2}`.
  - RLS: `tablePermission Sales = Sales[Region] = USERPRINCIPALNAME()`; `member 'x@co.com' = group`. OLS: `columnPermission Sales.'Unit Cost' = none`.
  - KPI block: `kpi { targetExpression / statusExpression / trendExpression / statusGraphic / trendGraphic }`.
  - createOrReplace recipes: add measures to existing table via `ref table Sales`, switch storage mode, create perspective.
- Source path: `plugins/powerbi-master/skills/tmdl-mastery/references/tmdl-examples-cookbook.md`.
- Quality: 5 — directly usable.
- Recommendation: adapt (use as canonical templates for our scaffold/calc-group/RLS skills; ensure dataset-agnostic — these use placeholder table/column names which is the right pattern; do NOT bake any specific field names into code).

### Four-layer validation taxonomy + Self-Validation Protocol → maps to pbi-model-doctor / pbi-report-reviewer agents + `modeling/bpa.ts`, `modeling/dax-reference-check.ts`, `visual/bind-validator.ts`
- What it is / why valuable: A clean mental model that mirrors and validates our own validator architecture: **(1) syntax/parser** (TMDL TmdlFormatException / PBIR JSON `$schema`), **(2) object/schema** (TmdlSerializationException / PBIR schemas), **(3) best-practice/BPA** (BPARules.json / PBI-InspectorV2), **(4) lineage/cross-reference** (TOM `model.Validate()` / bookmark→page + drillthrough + theme-resource checks). Plus a "validate-when-you-generate, never silently swallow errors, surface file/line/rule-id, distinguish static vs run-time" protocol. Confirms the design behind our model-reviewer/report-reviewer agents and gives explicit error-class → tool routing.
- Key content:
  - Error-class catalog (what each tool catches) and a "what validation CANNOT catch" list (credentials, refresh, DQ fallback, browser/mobile rendering) — good copy for our agent docs to set expectations.
  - The three TMDL exception types and why all three matter: `TmdlFormatException` (text→tokens), `TmdlSerializationException` (tokens→TOM), `model.Validate().Errors` (TOM semantic). Our `tmdl-parser`/`bpa` should produce analogous distinct error categories.
  - "Validation as a quality gate, not a ceremony" — only validate on generate/deploy/reported-failure, skip for conceptual/read questions. Good guidance for our hook + agent behavior.
- Source path: `plugins/powerbi-master/skills/validation-testing/SKILL.md` and `agents/powerbi-expert.md` (Self-Validation Protocol section).
- Quality: 5 — conceptually excellent and aligns with our build.
- Recommendation: adopt-as-is (as the framing/taxonomy for our reviewer agents and validator docs). The .NET/Python *implementations* are reference-only; the *taxonomy and error-routing* are language-agnostic.

### BPA rule catalog + rule-file format + severity strategy → maps to `modeling/bpa.ts` + bi-pattern-library + model-reviewer
- What it is / why valuable: Names ~60 standard Microsoft BPA rules by ID/severity/category with one-line "what it catches", documents the BPARules.json schema (ID/Name/Category/Severity 1-3/Scope/Expression/FixExpression), the constrained DAX-like expression language (TOM property predicates, `outerIt`, LINQ `Any/All`, `RegExMatch/Contains/Split`), Scope enum, and a CI severity strategy (3=Error blocks PR). This is a ready checklist of anti-patterns for our own BPA engine to implement natively in TS.
- Key content (high-signal rule IDs to mirror as TS checks):
  - Performance: `MODEL_PERFORMANCE_DISABLE_AUTO_DATETIME`, `..._AVOID_BIDIRECTIONAL_RELATIONSHIPS`, `DAX_PERFORMANCE_AVOID_DIVISION_OPERATOR` (use DIVIDE), `..._AVOID_FILTER_AS_FILTER_ARGUMENT`, `..._USE_VARIABLES`, `..._OPTIMIZE_RELATIONSHIP_COLUMNS_DATATYPE` (integer keys), `..._AVOID_FLOATING_POINT_DATATYPES`, `..._AVOID_HIGH_CARDINALITY_DATETIME_COLUMNS`.
  - Error prevention: `ERROR_PREVENTION_PROVIDE_FORMAT_STRING_FOR_MEASURES` (Error), `..._AVOID_INVALID_RELATIONSHIPS` (m2m), `..._USE_THE_DIVIDE_FUNCTION`.
  - DAX: `DAX_PRACTICE_AVOID_FORMAT_FUNCTIONS_IN_NUMERIC_MEASURES`, `..._USE_TREATAS_INSTEAD_OF_INTERSECT`, `..._USE_COALESCE_INSTEAD_OF_ISBLANK`, `..._AVOID_IFERROR`, `..._USE_SELECTEDVALUE_INSTEAD_OF_VALUES`.
  - Maintenance/Naming: `MAINTENANCE_HIDE_FOREIGN_KEYS`, `..._REMOVE_UNUSED_COLUMNS/MEASURES`, `..._PROVIDE_DESCRIPTION_FOR_MEASURES`, `NAMING_OBJECTS_SHOULD_NOT_START_WITH_SPACE` (Error), `..._NO_SNAKE_CASE_IN_NAMES`.
  - Rule-authoring tips: prefer positive predicates ("what should be true"), `or IsHidden` escape hatch, pin rule sets to a commit in CI.
- Source path: `plugins/powerbi-master/skills/validation-testing/references/bpa-rules-reference.md`.
- Quality: 5 — directly maps to BPA work.
- Recommendation: adapt (port the rule *intents* + severity tiers into our native `bpa.ts`; we don't need the Tabular Editor expression DSL, but the rule list is a high-quality coverage spec. Cross-check against what our `bpa.ts` already implements; this likely adds rules around floating-point/high-cardinality datetime/COALESCE/TREATAS).

### PBIR lineage cross-reference linter logic → maps to pbi-report-reviewer agent + a future PBIR lineage check (sibling of `visual/bind-validator.ts`)
- What it is / why valuable: Concrete algorithm (shown in Python, fully portable to TS) for the four most common PBIR breakages that JSON-schema validation can't catch: (1) bookmark `targetSection` → missing page, (2) drillthrough/tooltip `pageBindings.name` → missing page, (3) `report.json` `defaultPage` annotation → missing page, (4) theme `RegisteredResources` reference → missing static-resource file. Plus a structural validator (required entry files: `definition.pbir`, `report.json`, `version.json`, `pages/pages.json`; every page dir has `page.json`; every visual dir has `visual.json`; bookmark files end `.bookmark.json`) and a visual-type allowlist linter.
- Key content: build page/visual inventory by walking `definition/pages/*/page.json` + `.../visuals/*/visual.json`, then resolve every cross-reference against the inventory. Common PBIR failure table (e.g., `pageBinding.name` must be unique, never hand-edit `query.queryState`, theme RegisteredResource must be committed). Service limits to enforce: ≤1000 pages/report, ≤1000 visuals/page, ≤300 MB/report.
- Source path: `plugins/powerbi-master/skills/validation-testing/references/pbir-validation-recipes.md` (and the PBI-InspectorV2 rule-file format + starter rules in the same file).
- Quality: 5 — exactly the kind of check our report-reviewer should run.
- Recommendation: adapt (re-implement the lineage walker in TS as a report-level validator; the logic is language-agnostic. PBI-InspectorV2 itself is an external .NET tool = reference-only, but its rule shapes — `mustExist`, `isOneOf`, `arrayLengthLessThan`, `if/then` — are a good model for our rule predicates).

### PBIR JSON schema reference (visual/page/report structure) → maps to `visual/bind.ts`, `visual/bind-validator.ts`, `visual/templates.ts`, pbi-designer/report-builder + bi-pattern-library
- What it is / why valuable: The most complete PBIR file-format doc in the pack. Documents `definition.pbir` (byPath vs byConnection + the Fabric-REST `semanticmodelid` form), `page.json` (displayOption/visibility enums, height/width 1280×720), `visual.json` (`position` x/y/z/width/height/tabOrder, `visual.visualType`, `query.queryState`, `objects`, `drillFilterOtherVisuals`, `filters`), an extensive **visualType list** (~40 types), **query buckets by visual type**, `queryState` projections (`queryRef` like `Product.Category` / `Sum(Sales.Amount)`, `active`), the formatting `objects` model (`expr.Literal.Value` with `"true"`, `"14D"`, `"'#333333'"`, `solid.color`), and the categorical filter `Where/Condition/In/Expressions/Values` shape. Directly informs how we serialize/validate visual bindings.
- Key content (load-bearing details):
  - Query bucket map: Bar/Column → Category, Y, Series, Tooltips; Line → Category, Y, Series, Tooltips; Pie/Donut → Category, Y, Tooltips; Table → Values; Matrix → Rows, Columns, Values; Card → Fields; Slicer → Fields; Map → Location, Legend, Size, Tooltips; Scatter → X, Y, Size, Details, Legend. (Use this to validate that a binding targets a bucket that exists for the visual type — a natural extension to `bind-validator.ts`.)
  - Literal value encodings in `objects`: booleans/strings/numbers all wrapped as `{ "expr": { "Literal": { "Value": "..." } } }`; numeric suffix `D` (e.g., `"14D"`, `"0D"`); string literals single-quoted inside the value (`"'Sales by Category'"`).
  - `tableEx` = Table, `pivotTable` = Matrix, `clusteredBarChart`/`clusteredColumnChart` for clustered variants.
  - PBIP folder structure (2026): `*.Report/definition/{report.json,version.json,reportExtensions.json,pages/pages.json,pages/<p>/page.json,pages/<p>/visuals/<v>/visual.json,bookmarks/*.bookmark.json}`; `*.SemanticModel/definition/{database,model,relationships,expressions}.tmdl + tables/ roles/ cultures/ perspectives/`.
- Source path: `plugins/powerbi-master/skills/programmatic-development/references/pbir-schema-reference.md` and `skills/programmatic-development/SKILL.md` (PBIP structure, byPath/byConnection, PBIR limits/gotchas).
- Quality: 5 — high-fidelity PBIR reference.
- Recommendation: adopt-as-is (cross-check our visual templates and bind-validator against this — especially the query-bucket-per-visual-type map and the `expr.Literal.Value` encoding. The Python report-generator is reference-only; the schema details are directly load-bearing for our TS serializer).

## Cross-source overlap flags
- **DAX patterns / time-intelligence / function reference**: heavy overlap with the first-party / SQLBI-derived DAX content in the data-goblin sources and other mined repos. powerbi-master's DAX is correct but secondary — prefer first-party phrasings; use powerbi-master only to fill coverage gaps and for its anti-pattern table.
- **TMDL**: powerbi-master's TMDL *grammar/property reference* is arguably the cleanest in the whole corpus (the keyword→TOM-class + default-property-per-object tables). Where it overlaps the data-goblin dg3-semantic-models TMDL content, prefer powerbi-master for the *grammar spec* and the first-party repos for *real-world model examples / lineageTag conventions*.
- **PBIR schema**: overlaps the data-goblin dg1-pbip / dg2-reports PBIR content. The first-party repos likely have more real-report examples; powerbi-master wins on the *concise query-bucket-per-visual-type map* and the *formatting `expr.Literal.Value` encoding* — keep both.
- **BPA**: overlaps any source citing TabularEditor/BestPracticeRules. powerbi-master's contribution is the *consolidated rule-ID list with severities + the rule-file schema + severity-tier strategy* — a good single-page checklist; the rule set itself is the same upstream Microsoft one.
- **Validation taxonomy**: the four-layer model + PBIR lineage linter appear most crisply here; if other sources also describe validation, this one's *error-class→tool routing* and *static-vs-runtime* framing is the most reusable.
- **fabric-cicd / TOM / semantic-link-labs**: appears here and in Fabric-focused sources; all are non-Node runtimes (Python/.NET) — reference-only everywhere.
