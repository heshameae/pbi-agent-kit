# Mining findings: antigravity-awesome-skills — catalog/index/packaging
Source: antigravity-awesome-skills.xml

## Relevance summary
Highly relevant for our exact focus. This is an aggregator of ~1462 skills, so it has *solved the "catalog of N skills" problem at scale*: a JSON-Schema-validated discovery manifest (`skills_index.json`), a generated `CATALOG.md` with a 9-category taxonomy and a per-skill `Skill | Description | Tags | Triggers` table, a consistent SKILL.md frontmatter contract (`name/description/risk/source/date_added`), a repeatable description+body convention that drives auto-trigger discoverability, and two real Claude/Codex plugin-bundle manifests. The pack is a curated subset (~12 SKILL.md bodies + the index schema + full CATALOG.md + README + maintenance walkthrough), so the *index/packaging/taxonomy* signal is strong and concrete; the per-skill *content* is mostly unrelated domains (discarded). Note: the catalog/index/README/data files are all **generated artifacts** — the source of truth is per-skill frontmatter + a build pipeline (`npm run validate|index|catalog|chain`).

## High-value extractions

### Skills-index discovery manifest schema → maps to our skills catalog/index
- What it is: A draft-2020-12 JSON Schema for an **array** manifest (`skills_index.json`) that hosts read to discover skills *without* loading any SKILL.md. README states the contract explicitly: "This keeps discovery stable (`id`, `path`, metadata) while ensuring hosts only load `SKILL.md` for requested `@skill-id` values." This is the lazy-load pattern we want for 25+ skills (don't pay context for every skill, only route + load on demand).
- Key content (reusable shape): top-level `type: array`, each item requires:
  ```
  required: [ id, path, category, name, description, risk, source, date_added ]
  id          : string (min 1)            // == @skill-id used to invoke
  path        : string, pattern "^skills/"// folder location
  category    : string (min 1)            // taxonomy bucket (see categories below)
  name        : string (min 1)            // == folder name == frontmatter name
  description : string                     // auto-trigger text (also tokenized for search)
  risk        : string (min 1)            // unknown|none|safe|critical|offensive
  source      : string (min 1)            // community|official|<repo>
  date_added  : string|null
  plugin?     : { targets: {<tool>:<path>}, setup: {...}, reasons: [string] }  // optional per-tool install hints
  additionalProperties: true              // forward-compatible; extra keys allowed (author, tags, tools...)
  ```
- Source path within repo: `schemas/skills-index.v1.schema.json`; consumed via `skills_index.json` (canonical) + `data/skills_index.json` (mirror); documented under README "Stable Skills Manifest v1".
- Quality: 5 — clean, minimal, versioned (`.v1.`), `additionalProperties:true` for evolution.
- Recommendation: adapt. We have far fewer skills, but adopt the same idea: a small validated `skills-index.json` (or a typed TS const) keyed by `id`, carrying `category`, `description`, and a skill-class field. Replace `risk` (security-oriented, irrelevant for a single-vendor BI plugin) with a field meaningful to us, e.g. `kind: pipeline | shared-knowledge | crud`. Keep `id == name == folder`.

### CATALOG.md taxonomy + per-skill table → maps to how we categorize/index our skills
- What it is: A single generated `CATALOG.md` that buckets every skill into a flat one-level taxonomy, one `##` section per category with its count, and a 4-column table per skill. This is the human-browsable twin of the JSON index.
- Key content — the **9-category taxonomy** (count) used to organize 1462 skills:
  ```
  architecture (96) · business (82) · data-ai (282) · development (207) ·
  general (354) · infrastructure (134) · security (175) · testing (32) · workflow (100)
  ```
  Header: `# Skill Catalog` / `Generated at: <ISO>` / `Total skills: 1462`.
  Per-category table columns: `| Skill | Description | Tags | Triggers |`
  Example row:
  ```
  | `kpi-dashboard-design` | Comprehensive patterns for designing effective KPI dashboards... | kpi, dashboard | kpi, dashboard, designing, effective, key, performance, indicator, dashboards... |
  ```
  Mechanics worth copying: `Tags` = the skill's declared tag list; `Triggers` = `tags ∪ tokenized(description)` (lowercased keywords from the description) — i.e. **the description is the raw material for both search and auto-trigger keywords**. Descriptions are truncated with `...` in the table for scannability.
- Source path within repo: `CATALOG.md` (generated by `npm run catalog`).
- Quality: 4 — flat single-level taxonomy is easy to maintain but coarse at 1462 (the `general (354)` bucket is a catch-all smell). For our ~25 skills, the structure is ideal; the catch-all warning is the lesson.
- Recommendation: adapt. Use a small fixed taxonomy aligned to our skill classes rather than domains, e.g. `pipeline`, `shared-knowledge`, `crud` (matches our brief's grouping), optionally a `meta`/`workflow` bucket. Generate a CATALOG.md (or section in README) from the index so it never drifts. Avoid a catch-all category.

### SKILL.md frontmatter contract → maps to our skill metadata + index generation
- What it is: Every skill's `SKILL.md` opens with YAML frontmatter whose **required keys mirror the index schema** (so the index is generated from frontmatter, single source of truth). Optional keys add discovery/targeting metadata.
- Key content (observed across all sampled skills):
  ```
  ---
  name: <slug, == folder name>
  description: <1-2 sentences, see craft below>
  risk: unknown | none | safe | critical | offensive
  source: community | official | <repo>
  date_added: 'YYYY-MM-DD'
  # optional, seen in richer skills:
  author: <name>
  tags: [analytics, product, metrics, ...]
  tools: [claude-code, antigravity, cursor, gemini-cli, codex-cli]   # multi-host targeting
  version: '1.0.0'
  source_repo / source_type: <provenance>   # added during import normalization
  ---
  ```
  Risk distribution in sample: `unknown` (default for un-triaged), `none`, `safe`; walkthrough confirms full taxonomy `unknown|none|safe|critical|offensive`.
- Source paths: e.g. `skills/analytics-product/SKILL.md` (full optional set incl. tags/tools), `skills/kpi-dashboard-design/SKILL.md`, every `*/SKILL.md`.
- Quality: 5 — frontmatter-as-source-of-truth with generated index/catalog is the right architecture; `folder == name == id` invariant is enforced by validation.
- Recommendation: adapt. Define a strict frontmatter schema for our skills with required `name/description` and our own `kind` + `mcp_tools`/`targets` instead of `risk`. Generate the index/catalog from frontmatter; enforce `folder == name`. Keep frontmatter minimal — extra prose belongs in the body.

### Auto-trigger description-craft conventions → maps to our skill `description:` craft for reliable routing
- What it is: Repeatable conventions that make descriptions reliably auto-trigger and feed the Triggers/search index. This is the single most important lever for "skills that auto-trigger reliably."
- Key content (patterns observed):
  - **Length**: descriptions run ~82–200 chars; oversized ones are explicitly *shortened during import* "for validation" (walkthrough repeatedly notes "shortened oversized frontmatter descriptions"). There is a hard validation budget on description length. Keep ~1–2 sentences.
  - **Two-part shape**: `<what it does / capability statement>. Use when <trigger situation 1>, <situation 2>, ...` The trailing "Use when ..." clause is the deliberate trigger-phrase carrier. Real examples:
    - `Build production Apache Airflow DAGs with best practices for operators, sensors, testing, and deployment. Use when creating data pipelines, orchestrating workflows, or scheduling batch jobs.`
    - `Implement data quality validation with Great Expectations, dbt tests, and data contracts. Use when building data quality pipelines, implementing validation rules, or establishing data contracts.`
  - **Explicit invocation triggers** for command-style skills: some embed the literal user phrases, e.g. `This skill should be used when the user asks to "implement LLM-as-judge", "compare model outputs", "create evaluation rubrics"...` and `CRITICAL: Use for X. Triggers on: <comma-list of exact tokens/keywords>`. The "Triggers on:" keyword dump is a routing aid.
  - **Keyword density**: because Triggers = tags ∪ tokenized(description), packing concrete domain nouns (tool names, artifact names, verbs) into the description directly improves discovery. Vague descriptions ("provide guidance") tokenize to junk.
  - **Body skeleton reinforces routing**: nearly every SKILL.md body uses a fixed scaffold:
    ```
    ## Use this skill when      (or "## When to Use This Skill")
    - <bullet trigger conditions>
    ## Do not use this skill when  (or "## Do Not Use This Skill When")
    - <bullet anti-trigger / negative scope>
    ## Instructions
    - ...
    ```
    The paired *positive + negative* scope ("Do not use when... a simpler more specific tool can handle the request") is used to prevent over-triggering — directly relevant to a 25-skill catalog where skills can collide. Walkthrough even logs fixing discoverability by "tightening the frontmatter description and 'When to Use' guidance."
- Source paths: `skills/*/SKILL.md` frontmatter + bodies; CATALOG.md `data-ai`/`architecture` rows for the "Triggers on:" / "user asks to" idioms; `walkthrough.md` for the "shorten description / add When to Use" maintenance rules.
- Quality: 5 — this is battle-tested across 1462 skills and the maintenance log treats description+When-to-Use as *the* discoverability surface.
- Recommendation: adopt. Standardize our descriptions as `<capability>. Use when <2-4 concrete triggers>.`, cap length (~200 chars), pack BI nouns (TMDL, DAX, measure, visual binding, bookmark, theme, RLS, model). Add a `## Do not use when` negative-scope section to every SKILL.md to prevent cross-skill mis-routing — especially to disambiguate our shared-knowledge vs CRUD vs pipeline skills.

### Plugin-bundle packaging shape (.claude-plugin + .codex-plugin) → maps to our packaging
- What it is: "Bundles" are role-scoped sub-plugins, each a self-contained dir holding a `skills/` tree plus per-host manifests. Two manifest dialects ship side by side; only the manifest differs between hosts.
- Key content:
  - Directory shape:
    ```
    plugins/<bundle-name>/
      .claude-plugin/plugin.json     # Claude Code manifest
      .codex-plugin/plugin.json      # Codex manifest (richer)
      skills/<skill>/SKILL.md
      skills/<skill>/resources|references|examples/...
    ```
  - **Claude manifest** (`.claude-plugin/plugin.json`) — minimal:
    ```json
    { "name": "antigravity-bundle-data-engineering", "version": "11.4.1",
      "description": "...", "author": { "name": "...", "url": "..." },
      "homepage": "...", "repository": "...", "license": "MIT",
      "keywords": ["claude-code","skills","bundle","data-engineering","antigravity-awesome-skills"] }
    ```
  - **Codex manifest** (`.codex-plugin/plugin.json`) — adds `skills` dir pointer + an `interface` block for marketplace display:
    ```json
    { "name":"agyb-data-engineering", "version":"11.4.1", ...,
      "skills":"./skills/",
      "interface": {
        "displayName":"Data Engineering",
        "shortDescription":"Data & Analytics · 5 curated skills",
        "longDescription":"For building data pipelines. Covers Data Engineer, Airflow DAG Patterns, and 3 more skills.",
        "category":"Data & Analytics",
        "capabilities":["Interactive","Write"],
        "websiteURL":"...", "brandColor":"#111827" } }
    ```
  - Root-level distribution (from README/walkthrough, not in pack body): the *whole repo* is also exposed as one Claude plugin via root `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json` (install: `/plugin marketplace add sickn33/antigravity-awesome-skills`). Walkthrough 2026-03-18 records a real gotcha: marketplace entry `source` must be `"./"` not `"."` per Claude Code's relative-path schema — worth heeding for our manifest.
  - Versions are lockstepped across all manifests (`11.4.1`); `keywords` are SEO/discovery-tuned.
- Source paths: `plugins/antigravity-bundle-business-analyst/.claude-plugin/plugin.json` & `.codex-plugin/plugin.json`; `plugins/antigravity-bundle-data-engineering/...`; README "Stable Skills Manifest v1" / "Plugins"; walkthrough 2026-03-14 & 2026-03-18.
- Quality: 4 — clean separation of host manifests over a shared skills tree; the `interface` block is Codex-specific (not needed for our Claude-only plugin) but its `displayName/shortDescription/category` fields are a good model for human-facing metadata. We already ship `.claude-plugin/plugin.json`, so this confirms our packaging direction and the `source:"./"` caveat.
- Recommendation: reference-only for the Codex half; adopt the Claude manifest shape + `source:"./"` marketplace caveat; consider their "one repo = one plugin, skills grouped by category internally" model rather than many sub-bundles, given our smaller catalog.

### Build/index pipeline + generated-artifact contract → maps to keeping our catalog in sync
- What it is: The catalog/index/README/`data/*.json` are **derived**, regenerated from frontmatter by named scripts, with a frozen validation "warning budget" and a "PRs must be source-only (never commit generated files)" rule.
- Key content (reusable pipeline vocabulary): `npm run validate` (frontmatter/schema), `npm run validate:references` (no dangling links), `npm run index` (build `skills_index.json`), `npm run catalog` (build `CATALOG.md`), `npm run chain` (orchestrate the set), `npm run check:warning-budget` (freeze accepted warning count so quality can't silently regress). Single contract file `tools/config/generated-files.json` lists every derived artifact so CI/docs/scripts agree. Contributors add only `skills/<name>/SKILL.md`; CI regenerates the rest and runs an automated `skill-review` check on SKILL.md changes.
- Source paths: `walkthrough.md` (entire maintenance log); README "Contributing".
- Quality: 4 — strong, if heavyweight for our scale.
- Recommendation: adapt (lightweight). Keep one source of truth (frontmatter), add a small generator + a `validate` step (TS/zod) that fails CI on bad/oversized descriptions and on `folder != name`. Skip the warning-budget machinery initially.

## Cross-source overlap flags
- **Description-craft + When-to-Use scaffold**: expect heavy overlap with the dedicated skill-authoring repos (the `agent-skills-design-breadth` / `agent-skills-pbi-meta` siblings, and obra/superpowers which this repo credits as origin). This repo is the *aggregator's normalized form* of those conventions (it actively rewrites imported skills to the `description ≤ ~200 chars` + `## When to Use` + `## Do not use when` standard) — treat it as corroborating evidence of the convention, and defer to a dedicated skill-design source for the deep "why."
- **Plugin packaging vs marketplace repos**: the `claude-plugin-marketplace-structure` / `powerbi-agentic-plugins-structure` siblings will cover `.claude-plugin/plugin.json` + `marketplace.json` in more depth. This repo adds two concrete data points: (1) per-bundle vs root-level plugin trade-off, (2) the `source:"./"` marketplace-path gotcha. Cross-check the manifest field list against those repos before finalizing ours.
- **Index/manifest idea**: the "validated JSON manifest + lazy SKILL.md load" pattern likely recurs in other large skill libraries; this repo's `schemas/skills-index.v1.schema.json` is the most explicit, schema-backed version — prefer it as the canonical reference for our index shape.

## Discarded / not relevant
- **All per-skill domain content** (business-analyst, kpi-dashboard-design, market-sizing, startup-financial-modeling, airflow/dbt/embedding/vector-db, sql-pro, grafana, data-storytelling, data-quality-frameworks, analytics-product internals, etc.): unrelated domains and not Power-BI authoring; the sibling agent owns BI/data domain mining. We only extracted their *frontmatter/section-skeleton shape*, not their bodies.
- **`resources/implementation-playbook.md`, `examples/*.md`, `references/data-sources.md`**: confirm the convention that long content lives in sibling files referenced from SKILL.md (a packaging note, captured above), but their contents are domain-specific and discarded.
- **`risk` taxonomy as a security signal** (`critical/offensive`, "AUTHORIZED USE ONLY" auto-notices, security-findings triage in walkthrough): irrelevant to a single-vendor BI plugin; we only borrow the *idea* of a small enum metadata field, repurposed as `kind`.
- **README marketing surface** (38k-star badges, multi-tool install matrix for Cursor/Gemini/Kiro/OpenCode, buy-me-a-coffee, ~150 credits/contributor lists, star-history): aggregator go-to-market, not applicable to us. Kept only: the "Stable Skills Manifest v1" contract, the `--category/--risk/--tags` *selective-install* idea (an installer could let users pull a subset of our skills by `kind`), and the install-path/manifest caveats.
- **Maintenance-process minutiae** (PR squash/fork-workflow contracts, contributor-sync, release:prepare/publish, GitHub Pages web app): repo-ops, not plugin structure. Kept only the generated-artifact + validate/index/catalog pipeline vocabulary.
