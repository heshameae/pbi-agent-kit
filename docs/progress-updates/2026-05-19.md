## Goal

Stand up deterministic DAX-reference and bridge-binding gates fronting Microsoft's `powerbi-modeling` MCP **before** touching any agent or skill, so the safety layer doesn't depend on prompt fidelity. Everything Phase A landed is enforced at the Claude Code hook / engine level — not in agent instructions.

---

## What shipped

### Engine — `packages/core`

- **`modeling/dax-reference-check.ts`** — new `daxReferenceCheck(expression, model, { hostTable?, uncommittedMeasures? })`. Strips DAX strings + comments, validates qualified `Table[Field]` and bare `[Measure]` references against parsed TMDL, accepts measures still pending in the sidecar. Returns `{ valid, missing, ambiguous, unsupported }`.
- **`hooks/sidecar.ts`** — connection + uncommitted-measure registry helpers (`resolveSidecarRoot`, `upsertConnection`, `resolveConnection`, `readConnections`, `readUncommittedMeasures`, `appendUncommittedMeasures`, `clearUncommittedMeasures`). Atomic writes via `renameSync`. Root resolves `PBI_MCP_SIDECAR_DIR` → `${CLAUDE_PROJECT_DIR}/.pbi-mcp-ts/sidecar` → `${CLAUDE_PLUGIN_DATA}/sidecar`.
- **`modeling/tmdl-parser.ts`** — measures now carry `annotations: Record<string,string>`. New `unquoteAnnotationValue` handles real-TMDL double-quoted values with standard escape sequences. `unquoteIdent` left alone — identifier parsing unchanged.
- **`modeling/field-index.ts`** — `ModelMeasureField.annotations` propagated from `TMDLMeasure`.
- **`visual/bind-validator.ts`** — new `bridgeMetadataFor` parses `pbi_bridge_from/to/via/covers`; `isMetadataBridgeCoveredAxis` enforces them. Authoritative when annotations exist; expression-derived TREATAS scan is the fallback for legacy measures.

### MCP — `packages/mcp`

- New tool **`pbi_dax_reference_check`** (read-only) for ad-hoc validation against any `.SemanticModel/definition`.
- `buildServer()` factory exported so tests can drive the server via `InMemoryTransport` without spawning stdio; `main()` is gated so importing the module is side-effect-free.

### Hooks — `hooks/scripts/`

Each script carries a header explaining what it does and why (added 2026-05-19).

| Script | Event | Tool | Purpose |
| --- | --- | --- | --- |
| `gate-measure-create.mjs` | PreToolUse | `measure_operations` | Deny `Create` when DAX references something that exists neither in TMDL nor in the same-batch / sidecar uncommitted set. Fail-closed on missing connection or parse failure. |
| `update-connection-sidecar.mjs` | PostToolUse | `connection_operations` | Record every `ConnectFolder` in `sidecar/connections.json`; track lastUsed. |
| `track-uncommitted-measure.mjs` | PostToolUse | `measure_operations` | Append every successful `Create` to `sidecar/uncommitted-measures.json` so the gate can cross-reference them before Export. |
| `clear-uncommitted-on-export.mjs` | PostToolUse | `database_operations` | Clear one connection's pending set when `ExportToTmdlFolder` lands; resolves by `connectionName`, then `tmdlFolderPath`/`folderPath`. |

Shared helpers in `hooks/scripts/lib/`:

- `hook-io.mjs` — stdin JSON read, request shape normalization (handles both `tool_input.request` and direct `tool_input` forms), `allow()`/`deny()` exit contract.
- `core-import.mjs` — dynamic import of `packages/core/dist/index.js` from the hook scripts.

### Plumbing

- `hooks/hooks.json` — one PreToolUse + three PostToolUse entries for the modeling MCP. Matchers use an optional non-capturing group `mcp__(?:plugin_pbi-mcp-ts_)?…` so they fire on both standalone and plugin-namespaced tool surfaces.
- `.gitignore` — `.pbi-mcp-ts/` (project-scoped sidecar state).
- `.claude-plugin/plugin.json` — intentionally does **not** declare `hooks`; Claude Code auto-loads the standard `hooks/hooks.json` path and explicit declaration errors with "duplicate hooks file detected."

## What surfaced live (carry into Phase B)

1. **Plugin MCP tool names are namespaced.** Claude Code surfaces plugin-bundled MCP tools as `mcp__plugin_<plugin-id>_<server>__<tool>`. Initial matchers omitted the prefix and silently never fired. Matchers now tolerate both forms via `(?:plugin_pbi-mcp-ts_)?`.
2. **Manifest `hooks` declaration is a duplicate.** Claude Code auto-loads `hooks/hooks.json` from the standard path; explicit `"hooks": "./hooks/hooks.json"` errors. Plan's Stop-Line item "plugin manifest declares hooks" was inverted in this Claude Code version.
3. **ConnectFolder is offline-only.** Power BI Desktop does **not** watch TMDL files. Even after a clean Export, Desktop shows stale state until the `.pbip` is reopened. Live XMLA editing is a different MCP entry point — out of scope for Phase A.
4. **Architect skill silently rewrites user DAX.** A user-stated `SUM(col * 123)` became `SUMX(table, table[col] * 123)` without confirmation. The gate validates reference correctness, not expression fidelity to user intent. Phase B / skill-doc concern.
5. **Architect skill claims it can "bypass the check."** It cannot. PreToolUse hooks are enforced at the Claude Code runtime layer, not by skills. Skill prompt needs to be updated to remove the misleading offer.

## Reviewer-driven follow-ups landed before merge

The first verification pass surfaced four issues; all resolved before this update:

1. `clear-uncommitted-on-export.mjs` now reads `request.tmdlFolderPath ?? request.folderPath` — Microsoft's `ExportToTmdlFolder` actually emits `tmdlFolderPath`.
2. `gate-measure-create.mjs` now unions sidecar uncommitted measures with every `{table, name}` pair from the current Create batch, so forward-references inside one batch are allowed.
3. `tmdl-parser.ts` `unquoteAnnotationValue` correctly unwraps double-quoted annotation values with escape handling — covers real-TMDL emit where `annotation pbi_bridge_via = "TREATAS"` would otherwise compare as `"TREATAS"` against `TREATAS` and silently disable metadata enforcement.
4. `AGENTS.md` / `CLAUDE.md` confirmed intentional (one-line dataset-agnostic guardrails); kept.
